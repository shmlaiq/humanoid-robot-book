## 10.1 Introduction — The Edge Deployment Challenge

Building a humanoid robot perception pipeline in simulation is, by comparison, straightforward. The development workstation sits on a desk with a 300 W GPU, 64 GB of RAM, unlimited cooling airflow, and an Ethernet cable supplying uninterrupted network connectivity. The developer writes Python code, runs it, inspects the results in RViz2, and iterates. When the pipeline works, it feels complete.

Then comes the edge deployment challenge — and the developer discovers that the gap between "works on my laptop" and "works on the robot" is wider than any other gap in robotics engineering.

An AI humanoid robot carries its compute with it. The processing unit attached to the torso or shoulder must operate within a strict weight budget (every extra 100 g shifts the robot's center of mass), a strict power budget (every extra watt depletes the battery faster and requires more actuator work to stabilize thermally-induced vibrations), and a strict thermal budget (a module that throttles its clock speed when hot will stall the perception pipeline at the worst possible moment — in the middle of a task).

The transition from simulation to edge deployment forces a series of engineering decisions that compound in complexity: Which Jetson Orin module has enough compute without exceeding the power budget? Should ROS 2 nodes be cross-compiled for ARM or run inside Docker containers? How should PyTorch perception models be converted to TensorRT to run at acceptable latency on Orin's GPU? When the robot is operating at full load with all sensors active, what is the total power draw from the compute module, and does it fit within the power allocation from the battery management system?

This chapter answers these questions systematically. We begin with a detailed survey of the Jetson Orin product family, establishing the specifications and trade-offs of each module. We then walk through the complete JetPack SDK setup, the containerized deployment strategy using Docker on Jetson, the TensorRT optimization pipeline for production inference, and the thermal and power management strategies that keep the system stable under sustained load. The chapter concludes with a complete, end-to-end deployment example: a ROS 2 object detection node running TensorRT-optimized inference on a Jetson Orin NX, packaged in a Docker container, with a documented Dockerfile, launch file, and measured latency and power figures.

## 10.2 Jetson Orin Product Family

### 10.2.1 Orin Nano — Entry-Level Edge AI

The **Jetson Orin Nano** is NVIDIA's most accessible Orin-family module, targeting applications where cost and physical size are the primary constraints [1]. It is available in two memory configurations: 4 GB and 8 GB of LPDDR5 unified memory.

The Orin Nano 8GB delivers 40 TOPS (Tera Operations Per Second) of AI performance — a figure that, while modest by Orin family standards, substantially exceeds the previous-generation Jetson Nano (472 GOPS). It integrates a 6-core ARM Cortex-A78AE CPU, a 1024-core NVIDIA Ampere GPU with 32 Tensor Cores, and a single DLA core. The 8 GB memory configuration provides 68 GB/s of memory bandwidth, which is the primary determinant of throughput for memory-bound inference workloads.

For humanoid robots, the Orin Nano 8GB is practical for lightweight perception tasks — running a single object detection model at modest resolution, providing basic visual odometry — but is insufficient for the full multi-model, multi-camera perception stacks described in Chapter 9. It is best suited for task-specific satellite processors: handling a single camera feed dedicated to hand detection for manipulation tasks, or running a lightweight proximity alert system.

The Orin Nano 4GB variant, at 20 TOPS and 34 GB/s bandwidth, is primarily a cost-driven choice for applications with extremely limited perception requirements. Its 10 W maximum power draw makes it attractive for low-power sensor pods.

### 10.2.2 Orin NX — Mid-Range Performance

The **Jetson Orin NX** is the workhorse module for mid-tier humanoid robots and autonomous mobile platforms [1]. Available in 8 GB and 16 GB configurations, it shares the same physical form factor as the Orin Nano (69.6 mm × 45 mm) while delivering substantially higher performance.

The Orin NX 16GB achieves 100 TOPS and includes two DLA cores alongside the Ampere GPU, enabling parallelization between neural network inference on the DLA (for power-efficient sustained inference) and GPU compute for geometrically intensive tasks like stereo disparity and SLAM. The 16 GB memory configuration with 102 GB/s bandwidth supports concurrent multi-model inference without memory contention.

As established in Chapter 9's latency budget analysis, the Orin NX 16GB can sustain the Isaac ROS perception stack — VSLAM, object detection, and point cloud processing — for 640×480 stereo input at 30 Hz within a 50 ms latency budget and a 25 W power envelope. This combination makes it the practical choice for humanoid robots in the 15–25 kg total weight class.

The Orin NX 8GB (70 TOPS, 8 GB memory) provides a power-optimized alternative at approximately 10–20 W, accepting reduced parallel model capacity in exchange for a smaller battery drain.

### 10.2.3 AGX Orin — High-Performance Flagship

The **Jetson AGX Orin** is NVIDIA's highest-performance embedded AI module and the reference deployment target for demanding humanoid platforms [1]. Available in 32 GB and 64 GB configurations, it substantially exceeds the Orin NX in every performance dimension.

The AGX Orin uses the same Orin SoC as the NX but enables a larger power envelope (15–60 W configurable), unlocking higher GPU and CPU clock frequencies. The 2048-core Ampere GPU with 64 Tensor Cores and 204 GB/s memory bandwidth enables simultaneous execution of multiple high-resolution perception models. Two DLA cores can be loaded with fixed-function inference workloads (running continuously at ~5 W each) while the GPU handles variable-throughput tasks.

For humanoid platforms requiring multi-camera coverage (head, chest, hands), full-resolution (1080p) VSLAM, and simultaneous manipulation-specific perception (grasping point estimation, object 6-DoF pose estimation), the AGX Orin 32GB or 64GB is the appropriate choice. Its 100 mm × 87 mm form factor is larger than the NX module but remains embeddable in a torso compute bay.

## 10.3 JetPack SDK Setup

### 10.3.1 Flashing JetPack 6.x Using SDK Manager

JetPack 6.x (based on L4T R36) is the required software stack for all Jetson Orin modules [2]. Flashing from source using NVIDIA SDK Manager ensures a clean installation with all components correctly versioned and calibrated for the target hardware.

**Prerequisites on the host Ubuntu 22.04 machine:**

```bash
# Install SDK Manager from NVIDIA (requires NVIDIA Developer account)
# Download sdkmanager_*.deb from https://developer.nvidia.com/sdk-manager

sudo dpkg -i sdkmanager_2.2.0-12021_amd64.deb
sudo apt-get install -f    # Fix any dependency issues

# Connect Jetson AGX Orin devkit via USB-C to host machine
# Put devkit into Force Recovery Mode:
# 1. Hold FORCE RECOVERY button
# 2. Press and release RESET button
# 3. Release FORCE RECOVERY button

# Verify device is detected in recovery mode
lsusb | grep NVIDIA
# Expected: "Bus 001 Device 005: ID 0955:7023 NVIDIA Corp. APX"

# Launch SDK Manager
sdkmanager
```

In SDK Manager, select:
- **Product:** Jetson
- **Hardware:** Jetson AGX Orin Developer Kit (or appropriate module)
- **SDK Version:** JetPack 6.0 or newer
- **Components:** Jetson Linux, CUDA 12.x, cuDNN 8.x, TensorRT 8.x, VPI 3.x

The flash process takes approximately 20–40 minutes depending on host machine speed and selected components.

### 10.3.2 Post-Flash Verification

After flashing and first boot, verify the installation:

```bash
# Check L4T version
cat /etc/nv_tegra_release
# Expected: R36 (release), REVISION: 2.0

# Verify CUDA installation
nvcc --version
# Expected: Cuda compilation tools, release 12.x, V12.x.xxx

# Verify TensorRT installation
python3 -c "import tensorrt; print(tensorrt.__version__)"
# Expected: 8.x.x.x

# Check cuDNN
python3 -c "import ctypes; ctypes.CDLL('libcudnn.so.8')"
# No output = success

# Run hardware diagnostic
sudo tegrastats
```

### 10.3.3 Swap Memory and Power Mode Configuration

For a humanoid robot's primary compute module, configuring adequate swap memory prevents out-of-memory crashes during large model loading:

```bash
# Create 8 GB swap file (recommended for Orin NX 16GB)
sudo fallocate -l 8G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Verify swap
swapon --show
# Expected: NAME       TYPE SIZE USED PRIO
#           /swapfile  file   8G   0B   -2

# Persist swap across reboots
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Set maximum performance power mode (nvpmodel -m 0 = MAXN)
sudo nvpmodel -m 0
sudo jetson_clocks     # Lock CPU/GPU/EMC clocks to maximum frequency

# Verify current power mode
sudo nvpmodel -q
# Expected: NV Power Mode: MAXN
```

For mobile robot operation (battery-powered), the appropriate power mode depends on the mission phase:

```bash
# During active perception-heavy operation (navigation, manipulation)
sudo nvpmodel -m 0    # MAXN — all cores, maximum frequency

# During standby or slow walking
sudo nvpmodel -m 2    # 15W mode — reduced cores, lower frequency

# For ultra-low-power standby
sudo nvpmodel -m 3    # 10W mode
```

## 10.4 Deploying ROS 2 Nodes to Jetson

### 10.4.1 Cross-Compilation vs. Native Compilation

There are two strategies for building ROS 2 packages targeting Jetson Orin's ARM64 (aarch64) processor:

**Native compilation** builds the code directly on the Jetson device. This is the simpler approach — no cross-compilation toolchain configuration is needed — and it guarantees that the build environment perfectly matches the target runtime environment. The disadvantage is build time: a Jetson Orin NX 16GB with 8 CPU cores can build a moderately complex ROS 2 workspace in 10–30 minutes, compared to 2–5 minutes on a high-end x86_64 workstation.

**Cross-compilation** builds the code on an x86_64 development workstation using an ARM64 toolchain and sysroot. The output binary runs on the Jetson without modification. Cross-compilation is dramatically faster for large workspaces and enables automated CI/CD pipelines that produce Jetson-ready artifacts without requiring physical hardware. The disadvantage is configuration complexity: the ARM64 sysroot must match the JetPack version on the target Jetson, and any native library dependency (CUDA, TensorRT, cuDNN) must be available in the sysroot.

For most humanoid robot development workflows, **Docker-based deployment** (Section 10.4.2) provides a practical middle ground that avoids the complexity of native cross-compilation while still enabling efficient iteration.

### 10.4.2 Docker Containers on Jetson — NVIDIA L4T Base Images

NVIDIA provides official Docker base images for Jetson, built on **Linux for Tegra (L4T)**, that include CUDA, cuDNN, TensorRT, and VPI pre-installed for the Jetson Orin architecture [3]. These images are published on NVIDIA NGC (NVIDIA GPU Cloud) and Docker Hub under the `nvcr.io/nvidia/l4t-*` namespace.

Key base images for ROS 2 humanoid robot deployments:

| Base Image | Contents | Size |
|-----------|----------|------|
| `nvcr.io/nvidia/l4t-base:r36.2.0` | Ubuntu 22.04 + L4T drivers only | ~1.1 GB |
| `nvcr.io/nvidia/l4t-cuda:12.2.2-runtime` | L4T base + CUDA 12.x runtime | ~2.2 GB |
| `nvcr.io/nvidia/l4t-tensorrt:8.6.x` | CUDA + cuDNN + TensorRT 8.x | ~4.1 GB |
| `nvcr.io/nvidia/isaac/ros:aarch64-ros2_humble-3.1.0` | TensorRT + ROS 2 Humble + Isaac ROS | ~8.5 GB |

To run a GPU-enabled Docker container on Jetson, the NVIDIA Container Runtime must be configured:

```bash
# Install nvidia-container-runtime (typically pre-installed with JetPack 6.x)
sudo apt-get install -y nvidia-container-runtime

# Verify Docker can access GPU
docker run --rm --runtime nvidia \
    nvcr.io/nvidia/l4t-cuda:12.2.2-runtime \
    nvidia-smi
# Expected: Shows Jetson GPU information
```

### 10.4.3 Dockerfile for a ROS 2 Node on Jetson

The following Dockerfile demonstrates building a custom ROS 2 perception node for Jetson Orin, layered on the NVIDIA L4T TensorRT base image:

```dockerfile
# =============================================================================
# Dockerfile — ROS 2 Humble Object Detection Node for Jetson Orin
# Base: NVIDIA L4T TensorRT 8.6 (JetPack 6.x / aarch64)
# ROS 2: Humble Hawksbill
# Author: Muhammad Faisal Laiq Siddiqui
# =============================================================================

FROM nvcr.io/nvidia/l4t-tensorrt:8.6.2-runtime AS base

# --- System dependencies ---
ENV DEBIAN_FRONTEND=noninteractive
ENV LANG=en_US.UTF-8
ENV ROS_DISTRO=humble

RUN apt-get update && apt-get install -y --no-install-recommends \
    # Locale configuration
    locales \
    # Required for ROS 2 apt repo
    curl \
    gnupg2 \
    software-properties-common \
    # Python build tools
    python3-pip \
    python3-dev \
    # SSL certificates (required for Neon/external DB connections)
    ca-certificates \
    # Build tools for native compilation of ROS packages
    build-essential \
    cmake \
    git \
    && locale-gen en_US.UTF-8 \
    && rm -rf /var/lib/apt/lists/*

# --- ROS 2 Humble installation ---
RUN curl -sSL https://raw.githubusercontent.com/ros/rosdistro/master/ros.key \
        -o /usr/share/keyrings/ros-archive-keyring.gpg \
    && echo "deb [arch=arm64 signed-by=/usr/share/keyrings/ros-archive-keyring.gpg] \
        http://packages.ros.org/ros2/ubuntu jammy main" \
        > /etc/apt/sources.list.d/ros2.list \
    && apt-get update \
    && apt-get install -y --no-install-recommends \
        ros-humble-ros-base \
        ros-humble-vision-msgs \
        ros-humble-sensor-msgs \
        ros-humble-cv-bridge \
        python3-colcon-common-extensions \
        python3-rosdep \
    && rosdep init \
    && rm -rf /var/lib/apt/lists/*

# --- Python dependencies ---
RUN pip3 install --no-cache-dir \
    numpy==1.24.4 \
    opencv-python-headless==4.8.1.78 \
    pycuda==2024.1

# --- Build stage: compile custom ROS 2 packages ---
FROM base AS builder

WORKDIR /ros2_ws

# Copy the ROS 2 workspace source
COPY src/ src/

# Install rosdep dependencies for all packages
RUN . /opt/ros/humble/setup.sh \
    && rosdep update \
    && rosdep install --from-paths src --ignore-src -y

# Build workspace
RUN . /opt/ros/humble/setup.sh \
    && colcon build \
        --symlink-install \
        --cmake-args -DCMAKE_BUILD_TYPE=Release \
    && rm -rf build/ log/

# --- Runtime stage: minimal final image ---
FROM base AS runtime

WORKDIR /ros2_ws

# Copy the compiled install directory from builder stage
COPY --from=builder /ros2_ws/install/ install/

# Copy TensorRT engine files (pre-converted on host, architecture-specific)
COPY models/ /models/

# Create non-root user for the ROS node process
RUN useradd -m -u 1000 robot && chown -R robot:robot /ros2_ws /models
USER robot

# Source ROS 2 and the workspace on container start
ENV ROS_DOMAIN_ID=42
SHELL ["/bin/bash", "-c"]
ENTRYPOINT ["/bin/bash", "-c", \
    "source /opt/ros/humble/setup.bash && \
     source /ros2_ws/install/setup.bash && \
     exec \"$@\"", "--"]

CMD ["ros2", "launch", "robot_perception", "detection_trt.launch.py"]
```

### 10.4.4 Docker Compose for Multi-Container Robot Stack

A production humanoid robot software stack typically comprises multiple specialized containers communicating over a shared ROS 2 DDS domain. Docker Compose orchestrates this multi-container deployment:

```yaml
# docker-compose.robot.yaml
# Humanoid robot software stack — Jetson Orin NX 16GB
# ROS 2 Humble, ROS_DOMAIN_ID=42

version: '3.8'

x-ros-env: &ros-env
  ROS_DOMAIN_ID: "42"
  RMW_IMPLEMENTATION: rmw_cyclonedds_cpp    # Low-latency DDS middleware
  CYCLONEDDS_URI: /config/cyclonedds.xml

services:
  # ---- Camera driver (RealSense D435i) ----
  camera:
    image: ros:humble-ros-base
    runtime: nvidia                          # Access GPU for hardware-accelerated decoding
    devices:
      - /dev/video0:/dev/video0              # USB camera device
    privileged: false
    environment:
      <<: *ros-env
    volumes:
      - /tmp/.ros:/tmp/.ros
      - ./config:/config:ro
    command: >
      ros2 launch realsense2_camera rs_launch.py
      enable_gyro:=true enable_accel:=true
      unite_imu_method:=linear_interpolation
    restart: unless-stopped

  # ---- Visual SLAM (Isaac ROS cuVSLAM) ----
  vslam:
    image: nvcr.io/nvidia/isaac/ros:aarch64-ros2_humble-3.1.0
    runtime: nvidia
    environment:
      <<: *ros-env
    volumes:
      - /tmp/.ros:/tmp/.ros
      - ./config:/config:ro
    command: >
      ros2 launch isaac_ros_visual_slam isaac_ros_visual_slam.launch.py
    depends_on:
      - camera
    restart: unless-stopped

  # ---- Object detection (custom TensorRT node) ----
  detection:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime
    runtime: nvidia
    environment:
      <<: *ros-env
    volumes:
      - /tmp/.ros:/tmp/.ros
      - ./models:/models:ro
      - ./config:/config:ro
    shm_size: '512mb'                        # Shared memory for zero-copy image passing
    command: >
      ros2 launch robot_perception detection_trt.launch.py
    depends_on:
      - camera
    restart: unless-stopped

  # ---- Navigation (Nav2) ----
  navigation:
    image: ros:humble-ros-base
    runtime: nvidia
    environment:
      <<: *ros-env
    volumes:
      - /tmp/.ros:/tmp/.ros
      - ./config:/config:ro
      - ./maps:/maps:ro
    command: >
      ros2 launch nav2_bringup navigation_launch.py
      params_file:=/config/nav2_params.yaml
      use_sim_time:=false
    depends_on:
      - vslam
    restart: unless-stopped
```

## 10.5 ONNX Model Export and TensorRT Optimization

### 10.5.1 Exporting a PyTorch Model to ONNX

The path from a trained PyTorch model to deployment on Jetson Orin passes through two conversion steps: PyTorch to ONNX (Open Neural Network Exchange), then ONNX to TensorRT engine. ONNX serves as the intermediate representation: a graph-based format that is framework-independent and hardware-portable [4].

The following example exports a fine-tuned YOLOv8 obstacle detection model to ONNX:

```python
#!/usr/bin/env python3
"""
export_to_onnx.py
Exports a fine-tuned YOLOv8 PyTorch model to ONNX format
for subsequent TensorRT conversion on Jetson Orin.

Dependencies: torch>=2.0.0, ultralytics>=8.0.0
"""

import torch
import torch.onnx
from ultralytics import YOLO
import numpy as np
import os

# ---- Configuration ----
CHECKPOINT_PATH = '/models/yolov8n_obstacle_ft.pt'   # Fine-tuned checkpoint
ONNX_OUTPUT_PATH = '/models/yolov8n_obstacle.onnx'
INPUT_SHAPE = (1, 3, 640, 640)                        # batch=1, RGB, 640×640
OPSET_VERSION = 17                                     # ONNX opset (TensorRT 8.6 supports ≤17)

def export_model():
    """Export YOLOv8 fine-tuned model to ONNX with dynamic batch axis."""
    print(f'Loading checkpoint: {CHECKPOINT_PATH}')
    model = YOLO(CHECKPOINT_PATH)

    # Export via Ultralytics built-in ONNX export
    # This handles the complex YOLOv8 head operations correctly
    model.export(
        format='onnx',
        imgsz=640,
        half=False,          # Export in FP32 first; quantize separately
        opset=OPSET_VERSION,
        simplify=True,       # Run onnx-simplifier to reduce graph complexity
        dynamic=False,       # Fixed batch size for optimal TensorRT optimization
    )

    print(f'ONNX model exported to: {ONNX_OUTPUT_PATH}')
    verify_onnx(ONNX_OUTPUT_PATH)


def verify_onnx(onnx_path: str) -> None:
    """Run ONNX shape inference and graph check."""
    import onnx
    import onnxruntime as ort

    # Load and check ONNX model
    model = onnx.load(onnx_path)
    onnx.checker.check_model(model)
    print(f'ONNX model is valid. IR version: {model.ir_version}')

    # Run shape inference
    model = onnx.shape_inference.infer_shapes(model)

    # Print input/output shapes
    print('Inputs:')
    for inp in model.graph.input:
        shape = [d.dim_value for d in inp.type.tensor_type.shape.dim]
        print(f'  {inp.name}: {shape}')
    print('Outputs:')
    for out in model.graph.output:
        shape = [d.dim_value for d in out.type.tensor_type.shape.dim]
        print(f'  {out.name}: {shape}')

    # Quick inference test with ONNXRuntime (CPU)
    session = ort.InferenceSession(onnx_path, providers=['CPUExecutionProvider'])
    dummy = np.random.rand(*INPUT_SHAPE).astype(np.float32)
    outputs = session.run(None, {session.get_inputs()[0].name: dummy})
    print(f'ONNX inference test passed. Output shape: {outputs[0].shape}')


if __name__ == '__main__':
    export_model()
```

### 10.5.2 Converting ONNX to TensorRT Engine

TensorRT analyzes the ONNX computation graph, fuses compatible layers, selects optimal CUDA kernels for the target GPU architecture, and optionally quantizes weights to reduce precision. The result is a serialized **engine file** (.engine) that the TensorRT runtime can load and execute directly without re-optimizing [5].

```python
#!/usr/bin/env python3
"""
build_trt_engine.py
Converts an ONNX model to a TensorRT 8.x engine file.
Supports FP32, FP16, and INT8 (with calibration) precision modes.

Tested on: Jetson Orin NX 16GB, JetPack 6.0, TensorRT 8.6
"""

import tensorrt as trt
import numpy as np
import os
from pathlib import Path

# ---- TRT logger ----
TRT_LOGGER = trt.Logger(trt.Logger.WARNING)


def build_engine(
    onnx_path: str,
    engine_path: str,
    precision: str = 'fp16',
    max_workspace_gb: float = 2.0,
    calibrator=None
) -> None:
    """
    Build a TensorRT engine from an ONNX model file.

    Args:
        onnx_path: Path to the input .onnx file.
        engine_path: Path to write the output .engine file.
        precision: One of 'fp32', 'fp16', 'int8'.
        max_workspace_gb: Maximum GPU memory TRT may use for optimization (in GB).
        calibrator: IInt8Calibrator instance (required for int8 mode).
    """
    assert precision in ('fp32', 'fp16', 'int8'), f'Invalid precision: {precision}'
    assert Path(onnx_path).exists(), f'ONNX file not found: {onnx_path}'

    builder = trt.Builder(TRT_LOGGER)
    network_flags = 1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH)
    network = builder.create_network(network_flags)
    parser = trt.OnnxParser(network, TRT_LOGGER)

    # Parse ONNX model
    print(f'Parsing ONNX: {onnx_path}')
    with open(onnx_path, 'rb') as f:
        if not parser.parse(f.read()):
            for i in range(parser.num_errors):
                print(f'  ONNX parse error [{i}]: {parser.get_error(i)}')
            raise RuntimeError('ONNX parsing failed.')
    print(f'  Network inputs:  {network.num_inputs}')
    print(f'  Network outputs: {network.num_outputs}')
    print(f'  Network layers:  {network.num_layers}')

    # Configure builder
    config = builder.create_builder_config()
    config.set_memory_pool_limit(
        trt.MemoryPoolType.WORKSPACE,
        int(max_workspace_gb * (1 << 30))   # Convert GB to bytes
    )

    if precision == 'fp16':
        if not builder.platform_has_fast_fp16:
            print('  WARNING: Platform does not natively support FP16. Using FP32.')
        else:
            config.set_flag(trt.BuilderFlag.FP16)
            print('  Precision: FP16 enabled.')

    elif precision == 'int8':
        if not builder.platform_has_fast_int8:
            raise RuntimeError('Platform does not support INT8.')
        config.set_flag(trt.BuilderFlag.INT8)
        config.set_flag(trt.BuilderFlag.FP16)   # Use FP16 for layers not supported in INT8
        if calibrator is None:
            raise ValueError('INT8 precision requires an IInt8Calibrator instance.')
        config.int8_calibrator = calibrator
        print('  Precision: INT8 with FP16 fallback.')

    else:
        print('  Precision: FP32 (default).')

    # Build and serialize the engine
    print('Building TensorRT engine (this may take several minutes on Jetson)...')
    serialized_engine = builder.build_serialized_network(network, config)
    if serialized_engine is None:
        raise RuntimeError('Engine build failed.')

    # Write engine file
    with open(engine_path, 'wb') as f:
        f.write(serialized_engine)

    size_mb = os.path.getsize(engine_path) / (1024 * 1024)
    print(f'Engine saved to: {engine_path} ({size_mb:.1f} MB)')


if __name__ == '__main__':
    build_engine(
        onnx_path='/models/yolov8n_obstacle.onnx',
        engine_path='/models/yolov8n_obstacle_fp16.engine',
        precision='fp16',
        max_workspace_gb=2.0
    )
```

### 10.5.3 FP32 vs. FP16 vs. INT8 Precision Trade-offs

TensorRT supports three inference precisions, and the choice among them is one of the most consequential deployment decisions for a humanoid robot:

**FP32 (32-bit floating point):** Full IEEE 754 single precision. Highest numerical accuracy, largest model size, lowest throughput. Used when accuracy is paramount and hardware supports it, or when debugging precision-sensitive models.

**FP16 (16-bit floating point):** Half precision, natively supported by all Jetson Orin Ampere GPU Tensor Cores. Reduces model memory footprint by approximately 50% and typically doubles inference throughput compared to FP32 on Tensor Core-capable hardware. For most computer vision models — object detection, segmentation, depth estimation — FP16 produces negligible accuracy degradation (typically less than 0.5% mAP reduction) [5].

**INT8 (8-bit integer):** Quarter of the FP32 memory footprint. Requires a calibration dataset to determine the optimal quantization scale for each layer's activations. INT8 can achieve 1.5–2.5× higher throughput than FP16 on Jetson Orin hardware, but accuracy loss is model-dependent and must be validated carefully. For a safety-critical application like obstacle detection, INT8 quantization requires thorough evaluation across representative deployment scenarios.

**Table 10.2 — Inference Latency Comparison: PyTorch vs. ONNX Runtime vs. TensorRT on Jetson Orin NX 16GB**

| Framework | Precision | YOLOv8n (640×640) Latency | FPS | Memory (GPU) |
|-----------|-----------|--------------------------|-----|-------------|
| PyTorch (eager mode) | FP32 | ~85 ms | ~12 | ~1.2 GB |
| PyTorch (torch.compile) | FP32 | ~55 ms | ~18 | ~1.1 GB |
| ONNX Runtime (CUDA EP) | FP32 | ~38 ms | ~26 | ~0.9 GB |
| ONNX Runtime (CUDA EP) | FP16 | ~25 ms | ~40 | ~0.5 GB |
| TensorRT 8.x | FP32 | ~22 ms | ~45 | ~0.8 GB |
| TensorRT 8.x | FP16 | ~11 ms | ~91 | ~0.4 GB |
| TensorRT 8.x | INT8 | ~7 ms | ~143 | ~0.2 GB |

*Approximate values based on published TensorRT benchmarks and community measurements for Orin NX class hardware. Actual performance varies with model complexity, batch size, and system load [5].*

### 10.5.4 Python Code for TensorRT Inference

```python
#!/usr/bin/env python3
"""
trt_inference.py
Runs TensorRT inference on a Jetson Orin using a pre-built .engine file.
Integrates with ROS 2 as a standard Python callback.

TensorRT 8.6 / CUDA 12.x / Python 3.10
"""

import tensorrt as trt
import pycuda.driver as cuda
import pycuda.autoinit     # Initializes CUDA context automatically
import numpy as np
from typing import List, Tuple


TRT_LOGGER = trt.Logger(trt.Logger.WARNING)


class TRTInferenceEngine:
    """
    Wraps a TensorRT engine for easy inference in a ROS 2 node.
    Manages GPU memory allocation and host-device transfers.
    """

    def __init__(self, engine_path: str):
        """
        Load a serialized TensorRT engine from disk.

        Args:
            engine_path: Path to the .engine file built by build_trt_engine.py.
        """
        self._engine_path = engine_path
        self._engine = None
        self._context = None
        self._bindings = []
        self._host_inputs = []
        self._host_outputs = []
        self._cuda_inputs = []
        self._cuda_outputs = []
        self._stream = None
        self._load_engine()

    def _load_engine(self) -> None:
        """Deserialize engine and allocate I/O buffers."""
        runtime = trt.Runtime(TRT_LOGGER)
        with open(self._engine_path, 'rb') as f:
            serialized_engine = f.read()
        self._engine = runtime.deserialize_cuda_engine(serialized_engine)
        self._context = self._engine.create_execution_context()
        self._stream = cuda.Stream()

        # Allocate host and device buffers for each binding (input or output)
        for binding_idx in range(self._engine.num_io_tensors):
            binding_name = self._engine.get_tensor_name(binding_idx)
            binding_shape = self._engine.get_tensor_shape(binding_name)
            binding_dtype = trt.nptype(self._engine.get_tensor_dtype(binding_name))
            binding_mode = self._engine.get_tensor_mode(binding_name)

            size = int(np.prod(binding_shape))
            host_mem = cuda.pagelocked_empty(size, binding_dtype)
            device_mem = cuda.mem_alloc(host_mem.nbytes)
            self._bindings.append(int(device_mem))

            if binding_mode == trt.TensorIOMode.INPUT:
                self._host_inputs.append(host_mem)
                self._cuda_inputs.append(device_mem)
            else:
                self._host_outputs.append(host_mem)
                self._cuda_outputs.append(device_mem)

        print(f'TensorRT engine loaded: {self._engine_path}')
        print(f'  Inputs:  {len(self._host_inputs)}')
        print(f'  Outputs: {len(self._host_outputs)}')

    def infer(self, input_array: np.ndarray) -> List[np.ndarray]:
        """
        Run inference on a pre-processed input array.

        Args:
            input_array: Preprocessed image tensor, shape (1, 3, H, W), float32 or float16.

        Returns:
            List of numpy arrays, one per output binding.
        """
        # Copy input to pinned host memory, then to GPU
        np.copyto(self._host_inputs[0], input_array.ravel())
        cuda.memcpy_htod_async(self._cuda_inputs[0], self._host_inputs[0], self._stream)

        # Execute inference asynchronously on the CUDA stream
        self._context.execute_async_v2(
            bindings=self._bindings,
            stream_handle=self._stream.handle
        )

        # Copy output tensors from GPU to host
        results = []
        for host_out, cuda_out in zip(self._host_outputs, self._cuda_outputs):
            cuda.memcpy_dtoh_async(host_out, cuda_out, self._stream)
        self._stream.synchronize()   # Wait for all async operations to complete

        for host_out in self._host_outputs:
            results.append(host_out.copy())

        return results

    def preprocess_image(self, image_bgr: np.ndarray, target_size: Tuple[int, int]) -> np.ndarray:
        """
        Preprocess a BGR image for YOLOv8 TensorRT inference.

        Args:
            image_bgr: Input BGR image from OpenCV.
            target_size: (width, height) tuple for resizing.

        Returns:
            Preprocessed float32 array with shape (1, 3, H, W), values [0.0, 1.0].
        """
        import cv2
        w, h = target_size
        # Resize preserving aspect ratio with letterboxing
        resized = cv2.resize(image_bgr, (w, h), interpolation=cv2.INTER_LINEAR)
        # BGR → RGB, HWC → CHW, normalize to [0, 1]
        rgb = cv2.cvtColor(resized, cv2.COLOR_BGR2RGB)
        chw = np.transpose(rgb, (2, 0, 1)).astype(np.float32) / 255.0
        # Add batch dimension
        return np.expand_dims(chw, axis=0)

    def destroy(self) -> None:
        """Release GPU resources."""
        del self._context
        del self._engine
```

### 10.6.3 Thermal Throttling and Mitigation

Jetson Orin begins thermal throttling (automatically reducing clock speeds) when GPU temperature exceeds 85°C [1]. For a humanoid robot operating in a warm environment under sustained perception workloads, thermal management is not optional — it is a system reliability requirement.

**Active cooling:** The Jetson AGX Orin Developer Kit ships with an active fan; production modules use third-party carrier boards with custom heatsink and fan designs. For a humanoid robot torso, a heatsink with a 5 W blower fan can maintain Orin NX junction temperature below 70°C at 25 W power draw in a 25°C ambient environment.

**Workload scheduling:** Rather than running all perception models continuously, a humanoid robot can dynamically enable models based on task context:

```python
# Perception mode manager — reduces power during non-critical phases
class PerceptionModeManager:
    MODES = {
        'navigation': ['vslam', 'obstacle_detection', 'costmap'],
        'manipulation': ['object_pose', 'grasp_planning', 'depth'],
        'standby':      ['vslam'],    # Maintain localization only
    }
```
