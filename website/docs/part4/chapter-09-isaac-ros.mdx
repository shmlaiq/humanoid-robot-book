## 9.1 Introduction — The Perception Bottleneck in Humanoid Robotics

A humanoid robot standing in a hospital corridor faces a problem that would stagger any programmer encountering it for the first time. In a single second, the robot must ingest approximately 30 stereo image pairs from its head-mounted cameras, a 360-degree point cloud from its chest-mounted LiDAR, IMU data streaming at 400 Hz from its torso, and joint state feedback from 28 or more motor encoders — all simultaneously. It must identify the nurse walking toward it, estimate her trajectory, locate the medication cart to its right, re-localize itself relative to a map it built during the previous shift, and issue corrective torques to its leg actuators to avoid drifting into the corridor wall, all before taking its next step in approximately 40 milliseconds.

This is the perception bottleneck: the computational demand of processing real-world sensor data at the speed required for safe, reactive humanoid behavior.

The naive solution — running all perception algorithms on a general-purpose CPU — fails immediately. A stereo depth estimation pipeline running on a modern server CPU might achieve 5–8 frames per second. A real-time humanoid navigation stack requires 25–30 Hz minimum for safe obstacle avoidance. The arithmetic does not work. The bottleneck is not algorithmic sophistication; it is raw computational throughput, specifically the kind of massively parallel throughput that only a GPU can provide.

NVIDIA's answer to this challenge is **Isaac ROS**: a collection of hardware-accelerated, GPU-optimized ROS 2 packages that run perception algorithms at the speeds humanoid robots require [1]. Isaac ROS does not replace the ROS 2 framework that robotics engineers already know — it extends it. A developer who understands ROS 2 publishers, subscribers, and launch files can adopt Isaac ROS incrementally, replacing CPU-bound perception nodes with GPU-accelerated equivalents while keeping the rest of their software stack unchanged.

This chapter dissects Isaac ROS in depth. We examine the architectural foundations — GXF, Nitros, and the hardware targets — before exploring the three perception domains that matter most for humanoid operation: Visual SLAM for localization and mapping, object detection and segmentation for scene understanding, and 3D point cloud processing for geometry-aware navigation. We then address the deployment of these capabilities on NVIDIA Jetson Orin hardware, analyzing the performance, power, and thermal constraints that govern what is achievable on an edge device weighing less than a kilogram.

## 9.2 Isaac ROS Architecture

### 9.2.1 What Isaac ROS Is

Isaac ROS (Robot Operating System) is a curated set of ROS 2 packages, maintained by NVIDIA, that implement computer vision and perception algorithms accelerated on NVIDIA GPUs and DLA (Deep Learning Accelerator) units [1]. The project is publicly available on GitHub under the NVIDIA-ISAAC-ROS organization and targets ROS 2 Humble Hawksbill as its primary supported distribution, running on Ubuntu 22.04 LTS.

The packages cover a broad perception surface:

- **Visual SLAM** (`isaac_ros_visual_slam`) — camera-based localization and mapping
- **Object detection** (`isaac_ros_detectnet`) — bounding-box detection using DetectNet v2
- **Image segmentation** (`isaac_ros_image_segmentation`) — semantic pixel labeling
- **Stereo depth** (`isaac_ros_stereo_image_proc`) — disparity computation and point cloud generation
- **AprilTag detection** (`isaac_ros_apriltag`) — fiducial marker localization
- **Human pose estimation** (`isaac_ros_human_pose_estimation`) — skeleton keypoint detection
- **Depth segmentation** (`isaac_ros_depth_segmentation`) — ESS (Efficient Stereo Segmentation) neural network
- **nvblox** (`isaac_ros_nvblox`) — GPU-accelerated 3D scene reconstruction

What distinguishes Isaac ROS from vanilla ROS 2 packages is not the algorithms themselves — Visual SLAM and object detection algorithms existed before Isaac ROS — but the execution substrate. Every Isaac ROS package is built on top of GXF and communicates internally via Nitros, two architectural layers described in the following sections.

### 9.2.2 GXF — Graph Execution Framework

The **Graph Execution Framework (GXF)** is NVIDIA's runtime for building and executing computation graphs on heterogeneous hardware [2]. It provides the execution engine beneath every Isaac ROS node.

In a standard ROS 2 node, computation is driven by callbacks: a subscription callback fires when a message arrives on a topic, the callback executes on a CPU thread, and the result is published to another topic. This model works well for low-rate, compute-light operations. For high-throughput perception pipelines — where a new stereo pair arrives every 33 ms and must be processed in under 10 ms to leave headroom for downstream nodes — the callback overhead and memory copy costs become significant.

GXF solves this by treating computation as a directed acyclic graph (DAG) of **codelets** (lightweight compute units), **components** (configurations and data containers), and **entities** (collections of components). The graph is compiled at initialization time. At runtime, GXF schedules codelet execution across CPU threads and GPU streams according to data availability, minimizing latency by eliminating unnecessary synchronization points.

For a typical Isaac ROS visual perception pipeline, GXF manages:

1. **Image ingestion** — receiving raw bytes from the camera driver
2. **Format conversion** — converting from sensor_msgs/Image to a GPU-native NV12 or RGB format
3. **Neural network inference** — dispatching TensorRT inference on the GPU
4. **Post-processing** — non-maximum suppression, coordinate transformation
5. **Output publication** — writing results back to a ROS 2 topic

The entire sequence executes with minimal host-device memory transfers because GXF maintains tensor data in GPU memory throughout, using CUDA streams to overlap operations.

### 9.2.3 Nitros — NVIDIA Isaac Transport for ROS

Standard ROS 2 message passing over DDS (Data Distribution Service) involves serialization overhead that becomes costly for large messages. A single 1920×1080 RGB image serialized as a `sensor_msgs/Image` message occupies approximately 6.2 MB. At 30 Hz, this represents 186 MB/s of data that must be serialized, copied from GPU memory to CPU memory, transmitted over DDS, and then deserialized — before the receiving node can copy it back to GPU memory for processing.

**Nitros (NVIDIA Isaac Transport for ROS)** eliminates the bulk of this overhead through **zero-copy message passing** for GPU-resident data [3]. Instead of copying image data through the CPU, Nitros maintains the data in GPU memory and passes a handle (a lightweight reference structure containing a CUDA memory pointer and metadata) through the DDS layer. The receiving Isaac ROS node uses this handle to access the data directly from GPU memory without a host-device round trip.

This approach requires that both the publishing node and the subscribing node are Nitros-compatible — meaning both are built on GXF. When a Nitros-compatible node connects to a non-Nitros node (for example, a standard ROS 2 C++ node), Nitros automatically inserts a conversion node that materializes the handle into a standard ROS 2 message, preserving compatibility with the broader ROS 2 ecosystem.

The practical performance impact of Nitros is significant. In benchmarks published by NVIDIA for the Isaac ROS AprilTag package, Nitros-enabled pipelines achieve approximately 2× lower latency and 25–40% higher throughput compared to equivalent pipelines using standard ROS 2 message passing for HD resolution images [3].

**Figure 9.2 — Nitros Zero-Copy Message Passing**

```

       │                                       │
```

### 9.2.4 Supported Hardware

Isaac ROS is designed to run on three categories of hardware:

**NVIDIA Jetson Orin family** — the primary deployment target for mobile robots. The Jetson AGX Orin (32 GB and 64 GB variants), Jetson Orin NX (8 GB and 16 GB), and Jetson Orin Nano (4 GB and 8 GB) all support Isaac ROS. Each Jetson Orin SoC integrates a CPU (ARM Cortex-A78AE), an NVIDIA Ampere GPU, and one or two DLA cores, allowing perception workloads to be split between the GPU and the DLA based on latency and power requirements.

**Desktop NVIDIA GPU** — for development and testing on a workstation. Isaac ROS runs on Ubuntu 22.04 with any NVIDIA GPU that supports CUDA 11.4+ (Turing architecture or newer). Developers routinely prototype Isaac ROS pipelines on an RTX 3090 or RTX 4090 before cross-compiling or containerizing for Jetson deployment.

**DRIVE Orin** — NVIDIA's automotive-grade SoC, sharing the same architecture as Jetson AGX Orin but rated for automotive safety standards. While primarily targeted at autonomous vehicle applications, DRIVE Orin is occasionally used in high-end humanoid research platforms where automotive-grade reliability is required.

## 9.3 Visual SLAM with isaac_ros_visual_slam

### 9.3.1 What Visual SLAM Is

**SLAM (Simultaneous Localization and Mapping)** is the computational problem of building a map of an unknown environment while simultaneously determining the agent's location within that map [4]. For a humanoid robot operating in a building it has never entered, SLAM answers two questions at once: "Where am I?" and "What does this place look like?"

**Visual SLAM (VSLAM)** solves this problem using cameras as the primary sensor, in contrast to LiDAR SLAM which relies on laser ranging. The distinction matters enormously for humanoid robots because cameras are lightweight (a global-shutter stereo camera pair weighs approximately 50–100 g), power-efficient (5–15 W), and informationally rich — a camera captures texture, color, and appearance features that enable long-term place recognition, a capability LiDAR point clouds lack due to their structural sparsity in feature-poor environments.

Isaac ROS Visual SLAM (`isaac_ros_visual_slam`) implements **cuVSLAM**, NVIDIA's GPU-accelerated visual-inertial odometry (VIO) library [5]. The algorithm fuses stereo camera images and IMU measurements using a tightly coupled optimization backend, estimating the robot's 6-DoF pose (position and orientation in 3D space) at up to 60 Hz.

The cuVSLAM processing pipeline involves:

1. **Feature extraction** — detecting and describing keypoints (corners, blobs) in each camera frame using GPU-parallelized feature detectors
2. **Stereo matching** — establishing correspondences between left and right camera images to recover 3D point positions
3. **IMU pre-integration** — integrating IMU measurements between camera frames to predict motion
4. **Visual-inertial fusion** — optimizing a sliding-window factor graph combining visual reprojection constraints and IMU dynamics
5. **Loop closure detection** — recognizing previously visited locations to correct accumulated drift
6. **Map management** — maintaining a sparse 3D map of tracked landmarks for relocalization

### 9.3.2 Setup and Launch Commands

Isaac ROS Visual SLAM is distributed as a Docker container. The recommended deployment method uses the Isaac ROS Common dev container [1]:

```bash
# Pull the Isaac ROS common container (JetPack 6.x / CUDA 12.x compatible)
docker pull nvcr.io/nvidia/isaac/ros:aarch64-ros2_humble_nova_developer_kit-3.1.0

# Install the visual SLAM package inside the container
sudo apt-get install -y ros-humble-isaac-ros-visual-slam

# Launch the visual SLAM node with default configuration
ros2 launch isaac_ros_visual_slam isaac_ros_visual_slam.launch.py
```

For a system with a ZED 2i stereo camera, the launch file requires remapping the standard topic names:

```bash
ros2 launch isaac_ros_visual_slam isaac_ros_visual_slam.launch.py \

  image_right_topic:=/zed/zed_node/right/image_rect_color \
```

### 9.3.4 Python Launch File Example

The following Python launch file demonstrates a complete Visual SLAM launch configuration suitable for a humanoid robot equipped with a RealSense D435i RGB-D camera and an external IMU:

```python
# File: launch/vslam_realsense.launch.py
# Isaac ROS Visual SLAM launch for Intel RealSense D435i
# ROS 2 Humble / Isaac ROS 3.x

import launch
from launch_ros.actions import ComposableNodeContainer, Node
from launch_ros.descriptions import ComposableNode
from launch.substitutions import LaunchConfiguration
from launch.actions import DeclareLaunchArgument


def generate_launch_description():
    # --- Launch arguments ---
    use_imu_arg = DeclareLaunchArgument(
        'use_imu',
        default_value='True',
        description='Enable IMU fusion for improved accuracy'
    )

    # --- RealSense camera node ---
    # Publishes /camera/color/image_raw, /camera/depth/image_rect_raw,
    # /camera/imu topics.
    realsense_node = Node(
        package='realsense2_camera',
        executable='realsense2_camera_node',
        name='realsense',
        parameters=[{
            'enable_gyro': True,
            'enable_accel': True,
            'unite_imu_method': 'linear_interpolation',
            'enable_sync': True,
            'depth_module.profile': '640x480x30',
            'rgb_camera.profile': '640x480x30',
        }]
    )

    # --- Visual SLAM node (composable, runs in shared container for Nitros) ---
    visual_slam_node = ComposableNode(
        package='isaac_ros_visual_slam',
        plugin='nvidia::isaac_ros::visual_slam::VisualSlamNode',
        name='visual_slam',
        parameters=[{
            # Enable IMU fusion
            'enable_imu_fusion': LaunchConfiguration('use_imu'),
            # Rectified images are already undistorted — no additional rectification
            'rectified_images': True,
            # Denoise images on GPU before feature extraction
            'denoise_input_images': False,
            # Number of CPU threads for optimization backend
            'num_cameras': 2,
            # Map cell size in meters for memory-bounded map management
            'map_cell_size': 0.05,
            # Gravity direction in camera frame (z-down for standard mounting)
            'imu_params.gyroscope_noise_density': 0.0003394,
            'imu_params.accelerometer_noise_density': 0.02,
        }],
        remappings=[
            # Remap to RealSense topic names
            ('stereo_camera/left/image', '/camera/infra1/image_rect_raw'),
            ('stereo_camera/right/image', '/camera/infra2/image_rect_raw'),
            ('stereo_camera/left/camera_info', '/camera/infra1/camera_info'),
            ('stereo_camera/right/camera_info', '/camera/infra2/camera_info'),
            ('visual_slam/imu', '/camera/imu'),
        ]
    )

    # --- Composable node container (enables Nitros zero-copy within container) ---
    vslam_container = ComposableNodeContainer(
        name='vslam_container',
        namespace='',
        package='rclcpp_components',
        executable='component_container_mt',   # Multi-threaded container
        composable_node_descriptions=[visual_slam_node],
        output='screen'
    )

    return launch.LaunchDescription([
        use_imu_arg,
        realsense_node,
        vslam_container,
    ])
```

### 9.3.5 Accuracy Comparison: VSLAM vs. LiDAR SLAM for Indoor Environments

The choice between VSLAM and LiDAR SLAM for indoor humanoid navigation involves trade-offs across multiple dimensions. Table 9.1 summarizes the comparison based on published evaluation studies and NVIDIA benchmark data.

**Table 9.1 — Visual SLAM vs. LiDAR SLAM for Indoor Humanoid Navigation**

| Criterion | isaac_ros_visual_slam (cuVSLAM) | LiDAR SLAM (e.g., SLAM Toolbox + Nav2) |
|-----------|--------------------------------|----------------------------------------|
| Localization accuracy (ATE) | ~1–3 cm in well-textured indoor environments [5] | ~0.5–1.5 cm with mid-grade LiDAR [6] |
| Tracking frequency | Up to 60 Hz on Jetson AGX Orin | 10–20 Hz typical (LiDAR scan rate) |
| Failure modes | Texture-poor areas (white walls), rapid rotation, motion blur | Glass walls (transparent surfaces), dynamic objects |
| Sensor cost | ~$200–$600 for stereo camera | ~$500–$15,000 for 3D LiDAR |
| Sensor weight | 50–200 g | 250–900 g (mid-grade 3D LiDAR) |
| Power consumption | 3–8 W (camera + computation) | 8–18 W (LiDAR + computation) |
| Long-term drift | Higher (requires loop closure or GPS correction) | Lower (dense geometric features resist drift) |
| Performance under varying lighting | Degrades in darkness without IR illumination | Unaffected by lighting |

For humanoid robots intended for indoor operation under consistent lighting conditions — hospital corridors, warehouses, office buildings — cuVSLAM provides accuracy sufficient for navigation while maintaining the sensor weight and power advantages critical for whole-body power budgets. In environments with large glass-walled spaces or outdoor operation, sensor fusion combining VSLAM with a sparse LiDAR or radar is preferable.

## 9.4 Object Detection and Segmentation

### 9.4.1 isaac_ros_detectnet — Running DetectNet v2 on GPU

**`isaac_ros_detectnet`** is the Isaac ROS package that runs inference on DetectNet v2 models, a convolutional neural network architecture designed for efficient object detection [7]. The package accepts a TensorRT engine file as input and publishes `vision_msgs/Detection2DArray` messages containing bounding box coordinates, class labels, and confidence scores for each detected object.

DetectNet v2 is a single-shot detector that produces grid-based predictions: for each cell in a downsampled feature grid, it predicts the probability that an object center lies in that cell, along with the bounding box offsets. Compared to two-stage detectors like Faster R-CNN, DetectNet v2 trades some accuracy for substantially lower inference latency, making it well suited for real-time robot operation.

The package interfaces with the TensorRT runtime through GXF, enabling the neural network inference to execute entirely on the GPU without returning intermediate results to the CPU. Post-processing (confidence thresholding and non-maximum suppression) also runs as GXF codelets.

### 9.4.2 isaac_ros_image_segmentation — Semantic Segmentation with NVIDIA TAO Models

**`isaac_ros_image_segmentation`** provides semantic segmentation: assigning a class label to every pixel in an image [1]. This capability enables a humanoid robot to understand the semantic category of each region in its visual field — distinguishing floor from wall, person from furniture, obstacle from navigable space — at the pixel level rather than the coarser bounding-box level.

The package supports models from the **NVIDIA TAO Toolkit** (Train, Adapt, Optimize), NVIDIA's platform for fine-tuning pre-trained perception models on custom datasets [8]. The segmentation backbone is typically a lightweight encoder-decoder architecture (such as UNet with a MobileNet v2 or ResNet-18 encoder) that balances segmentation quality against inference speed.

### 9.4.4 ROS 2 Message Types for Detections


### 9.4.5 Python Subscriber for Detection Results

The following Python ROS 2 node demonstrates subscribing to detection results from `isaac_ros_detectnet` and performing secondary processing — in this case, estimating the distance to each detected person using the stereo depth topic:

```python
#!/usr/bin/env python3
"""
detection_distance_estimator.py
Subscribes to Isaac ROS DetectNet results and stereo depth map.
Estimates distance to detected persons for humanoid collision avoidance.

ROS 2 Humble / isaac_ros_detectnet 3.x
Author: Muhammad Faisal Laiq Siddiqui
"""

import rclpy
from rclpy.node import Node
from rclpy.callback_groups import ReentrantCallbackGroup
from rclpy.executors import MultiThreadedExecutor

from vision_msgs.msg import Detection2DArray
from sensor_msgs.msg import Image
from std_msgs.msg import Float32MultiArray
from message_filters import ApproximateTimeSynchronizer, Subscriber

import numpy as np
from cv_bridge import CvBridge


class DetectionDistanceEstimator(Node):
    """
    Fuses object detections with depth image to estimate obstacle distances.
    Uses ApproximateTimeSynchronizer to align detection and depth timestamps.
    """

    def __init__(self):
        super().__init__('detection_distance_estimator')

        # Allow concurrent callbacks (detections and depth may arrive simultaneously)
        self._cb_group = ReentrantCallbackGroup()

        # --- Synchronized subscribers for detections + depth ---
        self._det_sub = Subscriber(
            self,
            Detection2DArray,
            '/detectnet/detections',
            qos_profile=10
        )
        self._depth_sub = Subscriber(
            self,
            Image,
            '/stereo_camera/depth/image_rect_raw',
            qos_profile=10
        )

        # Synchronize with 50 ms tolerance (camera at 30 Hz → ~33 ms between frames)
        self._sync = ApproximateTimeSynchronizer(
            [self._det_sub, self._depth_sub],
            queue_size=10,
            slop=0.05
        )
        self._sync.registerCallback(self._synchronized_callback)

        # Publisher: distance to each detected object class
        self._dist_pub = self.create_publisher(
            Float32MultiArray,
            '/perception/obstacle_distances',
            10
        )

        self._bridge = CvBridge()
        self.get_logger().info('DetectionDistanceEstimator initialized.')

    def _synchronized_callback(
        self,
        detections_msg: Detection2DArray,
        depth_msg: Image
    ) -> None:
        """
        Process synchronized detections and depth image.
        For each detected bounding box, sample the median depth within
        the box region to estimate distance.
        """
        # Convert depth image from ROS message to numpy array
        # Depth image is typically 16UC1 (uint16, millimetres) or 32FC1 (float32, metres)
        try:
            if depth_msg.encoding == '16UC1':
                depth_array = self._bridge.imgmsg_to_cv2(depth_msg, '16UC1').astype(np.float32)
                depth_array /= 1000.0  # Convert mm to metres
            elif depth_msg.encoding == '32FC1':
                depth_array = self._bridge.imgmsg_to_cv2(depth_msg, '32FC1')
            else:
                self.get_logger().warn(f'Unsupported depth encoding: {depth_msg.encoding}')
                return
        except Exception as e:
            self.get_logger().error(f'Depth conversion failed: {e}')
            return

        img_h, img_w = depth_array.shape
        distances_out = Float32MultiArray()

        for detection in detections_msg.detections:
            if not detection.results:
                continue

            # Highest-confidence hypothesis
            best = max(detection.results, key=lambda r: r.score)
            class_id = best.id       # e.g., "person"
            confidence = best.score

            # Skip low-confidence detections
            if confidence < 0.40:
                continue

            # Extract bounding box pixel coordinates (clamped to image bounds)
            bbox = detection.bbox
            x1 = int(max(0, bbox.center.position.x - bbox.size_x / 2.0))
            y1 = int(max(0, bbox.center.position.y - bbox.size_y / 2.0))
            x2 = int(min(img_w - 1, bbox.center.position.x + bbox.size_x / 2.0))
            y2 = int(min(img_h - 1, bbox.center.position.y + bbox.size_y / 2.0))

            # Sample depth within the central 50% of the bounding box
            # (avoid edges where depth is often unreliable)
            x_margin = int((x2 - x1) * 0.25)
            y_margin = int((y2 - y1) * 0.25)
            roi_depth = depth_array[
                y1 + y_margin : y2 - y_margin,
                x1 + x_margin : x2 - x_margin
            ]

            # Filter invalid depth readings (0 = no measurement)
            valid_depths = roi_depth[roi_depth > 0.1]
            if valid_depths.size == 0:
                continue

            # Use median for robustness against outliers
            distance_m = float(np.median(valid_depths))
            distances_out.data.append(distance_m)

            self.get_logger().debug(
                f'Detected {class_id} (conf={confidence:.2f}) at {distance_m:.2f} m'
            )

        self._dist_pub.publish(distances_out)


def main(args=None):
    rclpy.init(args=args)
    node = DetectionDistanceEstimator()
    # MultiThreadedExecutor allows the synchronized callback to run
    # concurrently with any timer or service callbacks
    executor = MultiThreadedExecutor(num_threads=4)
    executor.add_node(node)
    try:
        executor.spin()
    finally:
        executor.shutdown()
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## 9.5 3D Point Cloud Processing

### 9.5.1 isaac_ros_stereo_image_proc — Generating Point Clouds from Stereo Cameras

**`isaac_ros_stereo_image_proc`** is the Isaac ROS replacement for the standard ROS 2 `stereo_image_proc` package [1]. It performs disparity estimation and point cloud generation entirely on the GPU, achieving substantially higher throughput than the CPU-based implementation.

The package contains two composable nodes:

- **`DisparityNode`** — computes a dense disparity map from the rectified left and right images using GPU-accelerated Semi-Global Block Matching (SGBM) or a learned stereo matching network
- **`PointCloudNode`** — reprojects the disparity map to a 3D point cloud in the `sensor_msgs/PointCloud2` format, using the camera calibration parameters

A complete stereo processing chain requires rectified images as input. Rectification corrects for lens distortion and aligns the epipolar lines horizontally, a prerequisite for stereo matching. If the input camera driver does not publish rectified images, the standard `image_proc` package or its Isaac ROS equivalent must be inserted upstream.

### 9.5.2 Filtering and Downsampling with PCL

Point clouds generated from stereo cameras at 30 Hz can contain 300,000–1,000,000 points per frame, depending on image resolution and scene geometry. Direct consumption of unfiltered point clouds by the navigation stack imposes unnecessary computational load. Three standard filtering operations reduce this overhead:

**Voxel grid downsampling** reduces point density by replacing all points within each voxel (a small cubic volume element) with their centroid. A 2 cm voxel grid typically reduces a 500,000-point cloud to fewer than 50,000 points while preserving the geometry relevant for navigation.

**PassThrough filter** removes points outside a specified range along a given axis — for example, removing all points more than 5 meters ahead (too far to affect immediate navigation) or closer than 20 cm (potentially on the robot's own body).

**StatisticalOutlierRemoval** identifies and removes points whose mean distance to their k-nearest neighbors exceeds a threshold by more than a specified number of standard deviations, eliminating isolated measurement noise.

```python
# File: point_cloud_filter_node.py
# Applies PCL-style filtering to stereo point cloud before passing to Nav2

import rclpy
from rclpy.node import Node
from sensor_msgs.msg import PointCloud2
import sensor_msgs_py.point_cloud2 as pc2
import numpy as np


class PointCloudFilter(Node):
    """
    Downsamples and filters a stereo point cloud before passing
    it to the Nav2 navigation stack.
    """

    def __init__(self):
        super().__init__('point_cloud_filter')

        # Declare configurable parameters
        self.declare_parameter('voxel_size', 0.02)       # 2 cm voxel grid
        self.declare_parameter('min_range', 0.25)         # Minimum distance in metres
        self.declare_parameter('max_range', 5.0)          # Maximum distance in metres

        self._voxel_size = self.get_parameter('voxel_size').value
        self._min_range = self.get_parameter('min_range').value
        self._max_range = self.get_parameter('max_range').value

        self._sub = self.create_subscription(
            PointCloud2,
            '/stereo/points2',
            self._cloud_callback,
            10
        )
        self._pub = self.create_publisher(
            PointCloud2,
            '/stereo/points2_filtered',
            10
        )

    def _cloud_callback(self, msg: PointCloud2) -> None:
        """Downsample and range-filter incoming point cloud."""
        # Read XYZ fields from PointCloud2 message
        points = np.array(
            list(pc2.read_points(msg, field_names=('x', 'y', 'z'), skip_nans=True))
        )

        if points.shape[0] == 0:
            return

        # PassThrough: keep only points within depth range (x-axis = forward)
        depth = np.sqrt(points[:, 0]**2 + points[:, 1]**2 + points[:, 2]**2)
        mask = (depth >= self._min_range) & (depth <= self._max_range)
        points = points[mask]

        if points.shape[0] == 0:
            return

        # Voxel grid downsampling via numpy binning
        vox = self._voxel_size
        bins = np.floor(points / vox).astype(int)
        # Use structured array to find unique voxels
        _, unique_idx = np.unique(bins, axis=0, return_index=True)
        points = points[unique_idx]

        # Publish filtered cloud
        header = msg.header
        filtered_msg = pc2.create_cloud_xyz32(header, points.tolist())
        self._pub.publish(filtered_msg)
        self.get_logger().debug(f'Published {points.shape[0]} filtered points.')
```

## 9.8 Chapter Summary and Bridge to Chapter 10

Hardware acceleration is not a luxury for humanoid perception — it is the enabling condition. A stereo depth pipeline running on a CPU stalls at 5 to 8 frames per second; the same pipeline running through Isaac ROS's GXF graph and Nitros zero-copy transport achieves 30 to 60 Hz on the Jetson Orin NX with headroom to spare. The architectural choices that make this possible — graph-based execution that avoids redundant CPU copies, Tensor Core-accelerated cuVSLAM that tracks pose at 60 Hz, TensorRT-optimized detection networks that identify objects in under 20 ms — are not implementation details. They are the reason a humanoid robot can perceive and act in real time on hardware that fits inside a torso panel.

Deploying these capabilities requires careful hardware selection within the Jetson Orin family, with the Orin NX 16GB representing a practical performance-power balance for mid-tier humanoid platforms and the AGX Orin serving demanding multi-camera, multi-model configurations. The latency budget analysis establishes that a 50 ms perception loop is achievable on Orin NX hardware with 640×480 stereo input — a result that satisfies the reactive requirements of a walking humanoid at pedestrian speeds.

The key insight from this chapter is that hardware acceleration is not a luxury for humanoid perception — it is a necessity. The gap between what a CPU-based system can deliver and what real-time embodied intelligence requires is not bridgeable with algorithmic optimization alone. It requires the GPU-accelerated, memory-efficient, pipeline-parallelized architecture that Isaac ROS provides.

Chapter 10 builds directly on this foundation. Once the perception pipeline is validated on a development workstation running Isaac ROS in a Docker container, the next engineering challenge is **deploying the entire robot software stack to the edge**: cross-compiling ROS 2 nodes for the ARM architecture, containerizing the stack with NVIDIA L4T base images, converting PyTorch perception models to TensorRT engines, and managing the power and thermal constraints of a mobile compute platform.
