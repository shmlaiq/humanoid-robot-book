## 6.1 Introduction — Why Simulation Is Essential for Humanoid Robotics

A humanoid robot falling on a hard floor is not an abstract problem. At roughly 50–80 kg of mass moving under gravity, an uncontrolled fall can bend actuators, crack structural composites, and destroy sensors worth thousands of dollars. More critically, if that fall happens in a hospital corridor or on a factory floor, it can injure a human being. This physical consequence is precisely why simulation occupies a foundational role in humanoid robotics engineering — consequences in the digital domain cost nothing except compute time.

The case for simulation goes well beyond accident avoidance. Consider the engineering timeline of a walking controller. A researcher developing a new gait algorithm needs to run thousands of iterations — adjusting PID gains, modifying foot placement heuristics, testing recovery behaviors after perturbation. On physical hardware, a single test cycle takes minutes to reset, requires a safety operator, and risks damage with every instability. In simulation, the same test cycle runs in seconds, the environment resets with a single command, and failure has no mechanical consequence. This speedup is not marginal; it is the difference between a six-month development cycle and a six-year one.

Simulation also enables systematic evaluation that is impossible on physical hardware. A walking controller must be tested on flat floors, inclines, loose gravel, stairs, and wet surfaces — the full distribution of environments a deployed robot will encounter. Constructing this test suite in the physical world is prohibitively expensive. In simulation, it is a configuration file.

**Gazebo Harmonic** — the current long-term support release of the open-source robotics simulator re-architected from the ground up beginning with Gazebo Garden [1] — is the environment that turns these advantages into practice. The stack built here defines a humanoid world in SDF, configures physics and sensor models, controls joints via plugins, and bridges the entire simulation to ROS 2. The result eliminates the cost and physical risk of hardware testing throughout the development cycles that follow.

## 6.2 Gazebo Harmonic Architecture

Gazebo Harmonic (released September 2023, supported through September 2028) represents the mature form of the "Ignition to Gazebo" re-architecture that began with Gazebo Citadel [2]. The monolithic simulator of the classic Gazebo era was decomposed into a family of loosely coupled libraries, each responsible for a single concern. Understanding this architecture is prerequisite to using the tool effectively.

### 6.2.1 gz-sim: The Simulation Engine

`gz-sim` is the core simulation library. It implements an **Entity Component System (ECS)** architecture — the same design pattern used in modern game engines [3]. In an ECS:

- **Entities** are lightweight identifiers (integers). A robot link, a joint, a sensor, and a world light are all entities.
- **Components** are plain data attached to entities. A `Pose` component holds position and orientation. A `JointVelocity` component holds velocity values.
- **Systems** are logic modules that iterate over entities possessing specific components and update their state.

This architecture has a critical property: systems are independent and can be loaded as plugins at runtime. Adding a new sensor type, a new controller, or a new physics integration does not require modifying the core simulator. It requires writing a new system plugin.

The simulation loop in `gz-sim` runs at a configurable step rate (default 1 kHz for robot simulation) and processes all loaded systems in a defined order: pre-update, update, post-update.

### 6.2.2 gz-transport: The Communication Middleware

`gz-transport` provides the publish-subscribe communication layer that connects simulation components, external controllers, and monitoring tools [4]. Built on ZeroMQ with Protobuf-serialized messages, it operates independently of ROS, allowing Gazebo to be used in non-ROS environments. Topics, services, and a parameter server are all provided by `gz-transport`.

Key message types relevant to humanoid simulation include:
- `gz.msgs.JointVelocity` — commanded velocity for each joint
- `gz.msgs.Pose` — position and orientation of any entity
- `gz.msgs.PointCloudPacked` — LiDAR scan data
- `gz.msgs.Image` — camera frame data
- `gz.msgs.IMU` — inertial measurement unit readings

### 6.2.3 gz-physics: Physics Engine Abstraction

`gz-physics` implements a plugin-based abstraction layer over multiple physics engines [5]. Gazebo Harmonic ships with support for two:

**DART (Dynamic Animation and Robotics Toolkit):** The default physics engine for Gazebo. DART implements Featherstone's articulated body algorithm for constraint-based rigid body dynamics, which provides high accuracy for jointed systems like robot arms and humanoid skeletons. It handles contact using a Velocity-based LCP (Linear Complementarity Problem) solver.

**Bullet Physics:** An open-source physics engine widely used in games and visual effects. Bullet provides an alternative for simulations requiring soft body dynamics or where DART's constraint solver produces instabilities in certain contact configurations.

The physics engine is selected in the world SDF file under the `<physics>` tag, allowing the same world file to be tested with different physics backends without code changes.

### 6.2.4 The Plugin System

Gazebo's plugin system is the mechanism by which simulation capabilities are extended. Plugins are shared libraries (`*.so` on Linux) loaded at runtime based on declarations in the SDF world or model file. The three most important plugin types for humanoid simulation are:

- **World plugins:** Loaded once at simulation start; implement world-level logic such as scenario scripting
- **Model plugins:** Attached to a specific robot model; implement controllers, sensor processors, or state monitors
- **Sensor plugins:** Attached to a sensor entity; implement data generation and publication

## 6.3 SDF World Building

SDF (Simulation Description Format) is an XML-based file format created specifically for robot simulation [6]. It extends URDF's capabilities with simulation-specific constructs: world-level physics configuration, multi-model environments, lighting, wind, atmospheric models, and plugin attachment points. Every Gazebo simulation begins with an SDF world file.

### 6.3.1 World File Structure

An SDF world file has the following top-level structure:

```xml
<?xml version="1.0" ?>
<sdf version="1.10">
  <world name="humanoid_world">

    <!-- Physics configuration -->
    <physics name="dart_physics" type="dart">
      <max_step_size>0.001</max_step_size>
      <real_time_factor>1.0</real_time_factor>
      <real_time_update_rate>1000</real_time_update_rate>
    </physics>

    <!-- Plugins loaded into the world -->
    <plugin
      filename="gz-sim-physics-system"
      name="gz::sim::systems::Physics">
    </plugin>
    <plugin
      filename="gz-sim-sensors-system"
      name="gz::sim::systems::Sensors">
      <render_engine>ogre2</render_engine>
    </plugin>
    <plugin
      filename="gz-sim-scene-broadcaster-system"
      name="gz::sim::systems::SceneBroadcaster">
    </plugin>
    <plugin
      filename="gz-sim-user-commands-system"
      name="gz::sim::systems::UserCommands">
    </plugin>

    <!-- Scene and lighting -->
    <!-- Ground plane -->
    <!-- Models (robot, objects) -->

  </world>
</sdf>
```

The `<physics>` block controls simulation accuracy and speed. `max_step_size` of 0.001 seconds (1 ms) is a standard value for robot simulation — small enough to capture fast joint dynamics but large enough to run near real time on modern hardware. The `real_time_factor` parameter controls simulation speed relative to wall clock time; values below 1.0 allow the simulation to run slower than real time when physics are computationally expensive.

### 6.3.2 Adding a Ground Plane, Lighting, and Gravity

A minimal viable world requires a ground plane for the robot to stand on, a directional light to enable rendering, and a gravity vector. Gazebo Harmonic defaults to Earth gravity (9.81 m/s² downward), but this can be overridden for lunar or zero-gravity simulations.

```xml
<!-- Gravity (default: Earth) -->
<gravity>0 0 -9.81</gravity>

<!-- Directional light (sun) -->
<light name="sun" type="directional">
  <cast_shadows>true</cast_shadows>
  <pose>0 0 10 0 0 0</pose>
  <diffuse>0.8 0.8 0.8 1</diffuse>
  <specular>0.2 0.2 0.2 1</specular>
  <direction>-0.5 0.1 -0.9</direction>
</light>

<!-- Ground plane model -->
<model name="ground_plane">
  <static>true</static>
  <link name="link">
    <collision name="collision">
      <geometry>
        <plane>
          <normal>0 0 1</normal>
          <size>100 100</size>
        </plane>
      </geometry>
      <surface>
        <friction>
          <ode>
            <mu>0.8</mu>
            <mu2>0.8</mu2>
          </ode>
        </friction>
      </surface>
    </collision>
    <visual name="visual">
      <geometry>
        <plane>
          <normal>0 0 1</normal>
          <size>100 100</size>
        </plane>
      </geometry>
      <material>
        <ambient>0.8 0.8 0.8 1</ambient>
        <diffuse>0.8 0.8 0.8 1</diffuse>
      </material>
    </visual>
  </link>
</model>
```

### 6.3.3 Importing a Humanoid Robot from URDF

Most humanoid robot descriptions are authored in URDF (Unified Robot Description Format), the ROS standard. Gazebo Harmonic provides the `gz-urdf` converter and a direct `<uri>` include mechanism. The recommended workflow for Gazebo Harmonic is to use the `sdformat` converter to produce native SDF from URDF, then include it in the world file.

Conversion at the command line:

```bash
gz sdf -p /path/to/robot.urdf > /path/to/robot.sdf
```

Including the converted robot in a world file:

```xml
<include>
  <uri>file:///path/to/robot/robot.sdf</uri>
  <name>humanoid</name>
  <pose>0 0 1.0 0 0 0</pose>
</include>
```

The `<pose>` element specifies the spawn position (x y z roll pitch yaw). A humanoid robot should be spawned with its feet at ground level — typically z = half the robot height above the floor.

The following is a complete, runnable world file for humanoid simulation:

## 6.4 Humanoid Robot Simulation

With the world defined, the robot model loaded, and physics configured, the next task is to make the robot move under control. Gazebo Harmonic provides a suite of joint control plugins that implement standard control laws without requiring external code.

### 6.4.1 Joint Control Plugins

**JointPositionController** implements a PD position controller for a single joint. It subscribes to a `gz.msgs.Double` topic containing the target angle and applies torque to drive the joint to that position.

```xml
<!-- Attach to the robot model in the SDF -->
<plugin filename="gz-sim-joint-position-controller-system"
        name="gz::sim::systems::JointPositionController">
  <joint_name>left_hip_pitch</joint_name>
  <topic>/humanoid/left_hip_pitch/cmd_pos</topic>
  <p_gain>100.0</p_gain>
  <i_gain>0.01</i_gain>
  <d_gain>10.0</d_gain>
  <i_max>1</i_max>
  <i_min>-1</i_min>
  <cmd_max>100</cmd_max>
  <cmd_min>-100</cmd_min>
</plugin>
```

**JointVelocityController** implements a PID velocity controller. This is the appropriate controller for wheel joints or joints where velocity, not position, is the commanded quantity.

```xml
<plugin filename="gz-sim-joint-velocity-controller-system"
        name="gz::sim::systems::JointVelocityController">
  <joint_name>left_wheel</joint_name>
  <topic>/humanoid/left_wheel/cmd_vel</topic>
  <p_gain>0.1</p_gain>
  <i_gain>0.0</i_gain>
  <d_gain>0.0</d_gain>
</plugin>
```

For a full humanoid with 30+ joints, each joint requires its own plugin block. This verbosity is typically managed by generating the SDF programmatically using Python's `xml.etree` or Jinja2 templating rather than hand-authoring every plugin declaration.

### 6.4.2 Contact Physics Configuration

The physical accuracy of foot contact simulation is critical for walking controllers. Three parameters dominate foot contact behavior:

**Friction coefficients (mu, mu2):** The Coulomb friction model uses two coefficients — `mu` (primary friction direction) and `mu2` (secondary friction direction). For a rubber sole on a concrete floor, values of 0.8–1.0 are physically realistic. Reducing these values simulates wet or icy surfaces.

**Restitution coefficient:** Controls the bounciness of a collision. For rigid robot feet on a hard floor, this should be near 0.0. A value of 0.0 means all kinetic energy in the collision normal direction is absorbed; the foot does not bounce.

**Contact stiffness (kp) and damping (kd):** These parameters govern the penalty-based contact model. Higher `kp` values make contacts stiffer (less penetration) but can cause numerical instability at large step sizes. A `kp` of 1×10¹³ with `kd` of 1 is appropriate for rigid body contact at 1 ms step size.

The following SDF fragment applies to a robot's foot link:

```xml
<collision name="left_foot_collision">
  <geometry>
    <box><size>0.22 0.08 0.02</size></box>
  </geometry>
  <surface>
    <friction>
      <ode>
        <mu>0.9</mu>
        <mu2>0.9</mu2>
      </ode>
      <bullet>
        <friction>0.9</friction>
        <friction2>0.9</friction2>
      </bullet>
    </friction>
    <bounce>
      <restitution_coefficient>0.0</restitution_coefficient>
      <threshold>1e6</threshold>
    </bounce>
    <contact>
      <ode>
        <kp>1e13</kp>
        <kd>1.0</kd>
        <max_vel>0.01</max_vel>
        <min_depth>0.001</min_depth>
      </ode>
    </contact>
  </surface>
</collision>
```

### 6.4.3 Simulating Foot Contact with the Floor

Foot contact sensing is required by walking controllers that need to know which feet are currently loaded. The `gz-sim-contact-system` plugin, loaded at the world level, enables contact reporting. Individual collision geometries must be marked as contact sensors:

```xml
<!-- In the foot link's collision block -->
<sensor name="left_foot_contact" type="contact">
  <contact>
    <collision>left_foot_collision</collision>
  </contact>
</sensor>
```

Contact events are published on the topic `/world/humanoid_world/model/humanoid/link/left_foot/sensor/left_foot_contact/contact`. A walking controller subscribes to this topic and uses binary contact state (contact / no contact) to implement phase detection for gait control.

## 6.5 Sensor Simulation

A critical advantage of high-fidelity simulators over simplified physics engines is accurate sensor simulation. Gazebo Harmonic simulates sensor noise models, mounting positions, and data formats matching real hardware — allowing sensor fusion code to run identically in simulation and on the physical robot.

### 6.5.1 LiDAR Sensor Plugin

A 3D rotating LiDAR (such as the Ouster OS1 or Velodyne VLP-16) provides the robot with a dense 3D point cloud of its environment. In simulation, the ray-cast sensor produces geometrically accurate returns with configurable noise.

```xml
<sensor name="lidar_top" type="gpu_lidar">
  <pose>0 0 0.6 0 0 0</pose>
  <update_rate>10</update_rate>
  <lidar>
    <scan>
      <horizontal>
        <samples>1800</samples>
        <resolution>1</resolution>
        <min_angle>-3.14159</min_angle>
        <max_angle>3.14159</max_angle>
      </horizontal>
      <vertical>
        <samples>16</samples>
        <resolution>1</resolution>
        <min_angle>-0.2618</min_angle>
        <max_angle>0.2618</max_angle>
      </vertical>
    </scan>
    <range>
      <min>0.1</min>
      <max>100</max>
      <resolution>0.01</resolution>
    </range>
    <noise>
      <type>gaussian</type>
      <mean>0.0</mean>
      <stddev>0.01</stddev>
    </noise>
  </lidar>
  <topic>/humanoid/lidar</topic>
  <gz_frame_id>lidar_link</gz_frame_id>
</sensor>
```

LiDAR data is published as `gz.msgs.PointCloudPacked` messages and, after bridging to ROS 2, appears as `sensor_msgs/PointCloud2` messages compatible with packages such as Nav2 and PCL.

### 6.5.2 RGB-D Camera Plugin

An RGB-D camera produces both a color image and a registered depth image. This is the primary sensor for object detection, grasp planning, and human detection in humanoid robots.

```xml
<sensor name="rgbd_camera" type="rgbd_camera">
  <pose>0.1 0 1.4 0 0.1 0</pose>
  <update_rate>30</update_rate>
  <camera name="head_camera">
    <horizontal_fov>1.047</horizontal_fov>
    <image>
      <width>640</width>
      <height>480</height>
      <format>R8G8B8</format>
    </image>
    <clip>
      <near>0.1</near>
      <far>10.0</far>
    </clip>
    <noise>
      <type>gaussian</type>
      <mean>0.0</mean>
      <stddev>0.007</stddev>
    </noise>
    <depth_camera>
      <output>depths</output>
      <clip>
        <near>0.1</near>
        <far>10.0</far>
      </clip>
    </depth_camera>
  </camera>
  <topic>/humanoid/head_camera</topic>
  <gz_frame_id>head_camera_link</gz_frame_id>
</sensor>
```

RGB images are published on `/humanoid/head_camera/image` and depth images on `/humanoid/head_camera/depth_image`, both as `gz.msgs.Image`. After bridging, these map to `sensor_msgs/Image` and work with standard ROS 2 camera drivers.

### 6.5.3 IMU Sensor Plugin

## 6.6 Integration with ROS 2 via ros_gz_bridge

Gazebo Harmonic and ROS 2 are independent middleware stacks with different message formats and communication protocols. The `ros_gz` package family provides the bridge that connects them, allowing Gazebo simulation data to flow into the standard ROS 2 ecosystem.

### 6.6.1 Installing ros_gz

For ROS 2 Humble on Ubuntu 22.04 with Gazebo Harmonic:

```bash
sudo apt-get update
sudo apt-get install ros-humble-ros-gz
```

This installs three packages:
- `ros_gz_bridge`: The topic bridge node
- `ros_gz_sim`: Launch utilities for starting Gazebo from ROS 2 launch files
- `ros_gz_image`: Optimized bridge for image topics using shared memory

### 6.6.2 Bridging Topics

The `ros_gz_bridge` node is a configurable bridge that maps Gazebo topics to ROS 2 topics with automatic message type conversion. It can be configured via command-line arguments or a YAML configuration file.

Bridge configuration YAML:

```yaml
# config/gz_bridge.yaml
- ros_topic_name: /humanoid/imu
  gz_topic_name: /humanoid/imu
  ros_type_name: sensor_msgs/msg/Imu
  gz_type_name: gz.msgs.IMU
  direction: GZ_TO_ROS

- ros_topic_name: /humanoid/lidar/points
  gz_topic_name: /humanoid/lidar
  ros_type_name: sensor_msgs/msg/PointCloud2
  gz_type_name: gz.msgs.PointCloudPacked
  direction: GZ_TO_ROS

- ros_topic_name: /humanoid/head_camera/image_raw
  gz_topic_name: /humanoid/head_camera/image
  ros_type_name: sensor_msgs/msg/Image
  gz_type_name: gz.msgs.Image
  direction: GZ_TO_ROS

- ros_topic_name: /humanoid/head_camera/depth
  gz_topic_name: /humanoid/head_camera/depth_image
  ros_type_name: sensor_msgs/msg/Image
  gz_type_name: gz.msgs.Image
  direction: GZ_TO_ROS

- ros_topic_name: /humanoid/joint_states
  gz_topic_name: /world/humanoid_world/model/humanoid/joint_state
  ros_type_name: sensor_msgs/msg/JointState
  gz_type_name: gz.msgs.Model
  direction: GZ_TO_ROS

- ros_topic_name: /humanoid/left_hip_pitch/cmd_pos
  gz_topic_name: /humanoid/left_hip_pitch/cmd_pos
  ros_type_name: std_msgs/msg/Float64
  gz_type_name: gz.msgs.Double
  direction: ROS_TO_GZ
```

The `direction` field controls data flow: `GZ_TO_ROS` for sensor data flowing out of simulation, `ROS_TO_GZ` for control commands flowing into simulation.

### 6.6.3 Python Launch File

The following ROS 2 Python launch file starts Gazebo Harmonic with the humanoid world, spawns the robot, and starts the topic bridge:

```python
# launch/humanoid_sim.launch.py
import os
from ament_index_python.packages import get_package_share_directory
from launch import LaunchDescription
from launch.actions import (
    DeclareLaunchArgument,
    IncludeLaunchDescription,
    ExecuteProcess,
    TimerAction,
)
from launch.launch_description_sources import PythonLaunchDescriptionSource
from launch.substitutions import LaunchConfiguration, PathJoinSubstitution
from launch_ros.actions import Node
from launch_ros.substitutions import FindPackageShare


def generate_launch_description():
    pkg_humanoid_sim = get_package_share_directory("humanoid_sim")
    pkg_ros_gz_sim = get_package_share_directory("ros_gz_sim")

    # Launch arguments
    use_gui = DeclareLaunchArgument(
        "use_gui",
        default_value="true",
        description="Start Gazebo with GUI (false for headless/CI)",
    )
    world_file = DeclareLaunchArgument(
        "world",
        default_value=os.path.join(pkg_humanoid_sim, "worlds", "humanoid_world.sdf"),
        description="Path to SDF world file",
    )

    # Start Gazebo
    gz_sim = IncludeLaunchDescription(
        PythonLaunchDescriptionSource(
            os.path.join(pkg_ros_gz_sim, "launch", "gz_sim.launch.py")
        ),
        launch_arguments={
            "gz_args": [
                LaunchConfiguration("world"),
                " -r",  # start running immediately
                " --headless-rendering"
                if LaunchConfiguration("use_gui") == "false"
                else "",
            ],
            "on_exit_shutdown": "true",
        }.items(),
    )

    # ros_gz_bridge node
    bridge = Node(
        package="ros_gz_bridge",
        executable="parameter_bridge",
        name="gz_bridge",
        parameters=[
            {
                "config_file": os.path.join(
                    pkg_humanoid_sim, "config", "gz_bridge.yaml"
                )
            }
        ],
        output="screen",
    )

    # Robot state publisher
    robot_state_publisher = Node(
        package="robot_state_publisher",
        executable="robot_state_publisher",
        name="robot_state_publisher",
        parameters=[
            {
                "robot_description": open(
                    os.path.join(
                        pkg_humanoid_sim, "urdf", "humanoid.urdf"
                    )
                ).read()
            }
        ],
        output="screen",
    )

    # RViz2 for visualization
    rviz = Node(
        package="rviz2",
        executable="rviz2",
        name="rviz2",
        arguments=[
            "-d",
            os.path.join(pkg_humanoid_sim, "rviz", "humanoid.rviz"),
        ],
        output="screen",
    )

    return LaunchDescription(
        [
            use_gui,
            world_file,
            gz_sim,
            # Delay bridge start to ensure Gazebo is ready
            TimerAction(period=3.0, actions=[bridge]),
            robot_state_publisher,
            TimerAction(period=4.0, actions=[rviz]),
        ]
    )
```

Launch the simulation with:

```bash
# With GUI
ros2 launch humanoid_sim humanoid_sim.launch.py

# Headless (no GUI window)
ros2 launch humanoid_sim humanoid_sim.launch.py use_gui:=false
```

## 6.7 Running Headless for CI

Continuous integration for robot software requires that simulation tests run automatically without a display. Gazebo Harmonic supports headless rendering using a virtual framebuffer, making it suitable for GitHub Actions and other CI environments.

### 6.7.1 GitHub Actions Workflow

```yaml
# .github/workflows/simulation_test.yml
name: Humanoid Simulation Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  sim-test:
    runs-on: ubuntu-22.04
    container:
      image: ros:humble-ros-base
      options: --privileged

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install Gazebo Harmonic
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            gz-harmonic \
            ros-humble-ros-gz \
            ros-humble-joint-state-publisher \
            ros-humble-robot-state-publisher \
            xvfb

      - name: Install Python dependencies
        run: |
          pip install pytest pytest-ros

      - name: Build ROS 2 workspace
        run: |
          source /opt/ros/humble/setup.bash
          cd /ros2_ws
          colcon build --symlink-install --packages-select humanoid_sim
          echo "source /ros2_ws/install/setup.bash" >> ~/.bashrc

      - name: Start virtual framebuffer
        run: |
          export DISPLAY=:99
          Xvfb :99 -screen 0 1280x1024x24 &
          sleep 2

      - name: Run simulation integration tests
        env:
          DISPLAY: ":99"
          RMW_IMPLEMENTATION: rmw_fastrtps_cpp
        run: |
          source /opt/ros/humble/setup.bash
          source /ros2_ws/install/setup.bash
          cd /ros2_ws
          python -m pytest \
            src/humanoid_sim/tests/test_simulation.py \
            -v --timeout=120
        timeout-minutes: 10

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: simulation-test-results
          path: /ros2_ws/test_results/
```

### 6.7.2 Automated Simulation Test Example

The following Python test launches the simulation headlessly, waits for topics to become active, and verifies that sensor data is being published at the expected rates:

```python
# tests/test_simulation.py
import pytest
import rclpy
import subprocess
import time
from rclpy.node import Node
from sensor_msgs.msg import Imu, PointCloud2, JointState


class SimulationTestNode(Node):
    def __init__(self):
        super().__init__("simulation_test_node")
        self.imu_received = False
        self.lidar_received = False
        self.joint_state_received = False
        self.imu_count = 0

        self.imu_sub = self.create_subscription(
            Imu, "/humanoid/imu", self._imu_cb, 10
        )
        self.lidar_sub = self.create_subscription(
            PointCloud2, "/humanoid/lidar/points", self._lidar_cb, 10
        )
        self.js_sub = self.create_subscription(
            JointState, "/humanoid/joint_states", self._js_cb, 10
        )

    def _imu_cb(self, msg):
        self.imu_received = True
        self.imu_count += 1

    def _lidar_cb(self, msg):
        self.lidar_received = True

    def _js_cb(self, msg):
        self.joint_state_received = True


@pytest.fixture(scope="module")
def ros_context():
    rclpy.init()
    yield
    rclpy.shutdown()


@pytest.fixture(scope="module")
def gazebo_process():
    """Start Gazebo headlessly for the test session."""
    proc = subprocess.Popen(
        [
            "ros2", "launch", "humanoid_sim",
            "humanoid_sim.launch.py",
            "use_gui:=false",
        ],
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
    )
    # Allow simulation to fully initialize
    time.sleep(15)
    yield proc
    proc.terminate()
    proc.wait(timeout=10)


def test_imu_publishing(ros_context, gazebo_process):
    """Verify IMU sensor publishes data."""
    node = SimulationTestNode()
    start = time.time()
    while not node.imu_received and (time.time() - start) < 10.0:
        rclpy.spin_once(node, timeout_sec=0.1)
    assert node.imu_received, "IMU topic not publishing within 10 seconds"
    node.destroy_node()


def test_lidar_publishing(ros_context, gazebo_process):
    """Verify LiDAR sensor publishes data."""
    node = SimulationTestNode()
    start = time.time()
    while not node.lidar_received and (time.time() - start) < 10.0:
        rclpy.spin_once(node, timeout_sec=0.1)
    assert node.lidar_received, "LiDAR topic not publishing within 10 seconds"
    node.destroy_node()


def test_joint_states_publishing(ros_context, gazebo_process):
    """Verify joint state publisher is active."""
    node = SimulationTestNode()
    start = time.time()
    while not node.joint_state_received and (time.time() - start) < 10.0:
        rclpy.spin_once(node, timeout_sec=0.1)
    assert node.joint_state_received, "Joint states not publishing within 10 seconds"
    node.destroy_node()


def test_imu_publish_rate(ros_context, gazebo_process):
    """Verify IMU publishes at approximately 200 Hz."""
    node = SimulationTestNode()
    # Collect data for 1 second
    start = time.time()
    while (time.time() - start) < 1.0:
        rclpy.spin_once(node, timeout_sec=0.005)
    # Allow wide tolerance for CI environment timing
    assert node.imu_count > 100, (
        f"IMU publish rate too low: {node.imu_count} messages in 1 second"
    )
    node.destroy_node()
```

*Figure 6.2: Four-simulator comparison positioned on physics accuracy vs. rendering quality axes. Isaac Sim leads on both dimensions (RTX rendering + PhysX 5 physics) but requires NVIDIA RTX hardware. MuJoCo leads on physics precision for contact-rich tasks. Gazebo leads on ROS 2 integration and accessibility. (* Unity free tier; ** Isaac Sim free for most uses.)*

The following analysis compares Gazebo Harmonic against the three other simulators most commonly used in humanoid robotics: Unity (Chapter 7), NVIDIA Isaac Sim (Chapter 8), and MuJoCo.

**Advantages of Gazebo Harmonic:**

*Open source with no licensing cost.* Gazebo is BSD-licensed. For universities, open research projects, and startups, there is no licensing barrier. This contrasts with Isaac Sim (requires NVIDIA hardware and an EULA) and Unity (commercial licensing for production use).

*Native ROS 2 integration.* The `ros_gz` ecosystem is maintained by Open Robotics alongside Gazebo itself, ensuring tight version coupling and reliable bridging. No other simulator has this level of native ROS 2 support.

*Physics accuracy for rigid body robotics.* DART's implementation of Featherstone dynamics provides high accuracy for articulated robot simulation. Contact simulation with the ODE/Bullet collision detector is well-validated against physical hardware [7].

*Extensible plugin architecture.* The ECS plugin system allows simulation capabilities to be added without modifying the simulator core. Custom sensor models, novel actuator types, and scenario-specific physics can all be implemented as plugins.

*SDF standardization.* SDF is a community standard with a formal specification and wide tooling support. Robot descriptions authored in SDF are portable across simulators that support the format.

**Disadvantages of Gazebo Harmonic:**

*Visual fidelity limitations.* Gazebo's renderer (Ogre2) produces visually adequate but not photorealistic scenes. Training computer vision perception algorithms in Gazebo and deploying to physical hardware often requires domain randomization or adaptation to bridge the visual sim-to-real gap.

*Performance at scale.* Simulating many robots simultaneously (a warehouse with 50 humanoids, for example) is computationally expensive in Gazebo because the physics engine does not parallelize across multiple CPU cores in the default configuration.

*Steep learning curve for world building.* Authoring complex environments in SDF XML is verbose and error-prone. Other simulators (Isaac Sim, Unity) provide GUI editors that accelerate scene construction.

*Sensor simulation completeness.* While Gazebo's IMU and LiDAR simulations are mature, simulation of more complex sensors (structured light depth cameras, tactile sensors, force-torque sensors with realistic noise models) requires custom plugin development.

*Documentation fragmentation.* The transition from classic Gazebo to Gazebo Harmonic left significant documentation gaps. Many online tutorials and community answers still refer to the classic API, causing confusion for new users.

## 6.9 Chapter Summary and Bridge to Chapter 7

Gazebo Harmonic occupies a specific and irreplaceable position in the humanoid robotics toolchain: it is the environment where controllers are tested without hardware risk, where sensor fusion algorithms are validated against synthetic ground truth, and where CI pipelines run automated regression suites that catch regressions before they reach the robot. The ECS architecture of `gz-sim`, the publish-subscribe messaging of `gz-transport`, and the pluggable physics backend of `gz-physics` together make this possible without binding the engineer to a single simulation framework or sensor model.

Gazebo's strengths — open licensing, ROS 2 integration, extensibility, and physics accuracy — make it the correct choice for the majority of humanoid robotics development tasks: controller development, navigation testing, sensor fusion development, and CI-driven regression testing.

However, Gazebo has limits. When the task demands photorealistic rendering — when the robot must interact with realistic human avatars, or when perception algorithms must train on visually authentic scenes — a different tool is required. The next chapter examines **Unity**, a game engine that brings commercial-quality rendering and behavioral simulation to humanoid robotics, enabling a class of Human-Robot Interaction experiments that Gazebo cannot support.
