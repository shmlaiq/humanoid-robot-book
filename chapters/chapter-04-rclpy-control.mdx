Chapter 3 established the architectural foundation: DDS-based communication, the four primitives (topics, services, actions, parameters), lifecycle nodes, and the workspace build system. That foundation is necessary, but it is not sufficient. Building software that actually controls a humanoid robot — with 28 to 44 degrees of freedom moving simultaneously, real-time force feedback from hands, and navigation through a hospital corridor — requires a deeper layer of engineering.

In this chapter, we transition from theory to practice. We will write Python nodes with proper type annotations, implement multi-threaded execution strategies to prevent callback starvation, integrate with the `ros2_control` framework that manages hardware interfaces, author a complete URDF (Unified Robot Description Format) description for a simplified humanoid arm, and connect motion planning through MoveIt 2. Every code block in this chapter is written to ROS 2 Humble standards [1], targeting Python 3.10+ with full type annotations. The examples are designed to be runnable in a real workspace with the dependencies installed in Chapter 3.

A clarifying note on scope: this chapter covers the software layer that commands a robot's joints, plans motions, and coordinates behavior. It does not cover the mechanical design of the joints themselves, the power electronics, or the actuator firmware — those subjects belong to robotics hardware engineering texts. Our focus is the software that bridges between high-level task commands and the hardware abstraction layer provided by `ros2_control`.

### 4.2.1 The Joint State Monitor Node

The following is a complete, production-quality node that subscribes to joint states, tracks joint positions over time, and publishes a simple status message. It demonstrates the structural patterns we will use throughout the chapter.

```python
# joint_state_monitor.py
# Monitors joint states for a humanoid arm and publishes health status.
# ROS 2 Humble | Python 3.10+ | rclpy
#
# Installation:
#   sudo apt install ros-humble-sensor-msgs ros-humble-std-msgs
#   colcon build --packages-select humanoid_arm_controller

from __future__ import annotations

import math
from typing import Optional

import rclpy
from rclpy.node import Node
from rclpy.qos import QoSProfile, ReliabilityPolicy, DurabilityPolicy, HistoryPolicy
from sensor_msgs.msg import JointState
from std_msgs.msg import String


# Type alias for joint position dictionary: maps joint name to position in radians
JointPositionMap = dict[str, float]

# Joint position limits for a simplified 3-DOF humanoid arm (in radians)
# These limits are illustrative; real limits come from URDF joint tags.
JOINT_LIMITS: dict[str, tuple[float, float]] = {
    'shoulder_pitch': (-2.617, 2.617),   # ±150 degrees
    'elbow_flex':     (-0.087, 2.967),   # -5 to +170 degrees
    'wrist_roll':     (-3.141, 3.141),   # ±180 degrees
}


class JointStateMonitor(Node):
    """
    Monitors joint states published on /joint_states and:
      1. Detects joints approaching their position limits (within 10 degrees)
      2. Publishes a warning string on /arm/health_status when a limit is approached
      3. Logs message receipt rate every 5 seconds

    Subscriptions:
      /joint_states  (sensor_msgs/JointState) — joint positions, velocities, efforts

    Publications:
      /arm/health_status  (std_msgs/String) — health warnings, JSON-like format
    """

    def __init__(self) -> None:
        super().__init__('joint_state_monitor')

        # --- QoS profiles ---
        # Joint state topic: high frequency, best-effort is acceptable
        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            durability=DurabilityPolicy.VOLATILE,
            history=HistoryPolicy.KEEP_LAST,
            depth=10,
        )

        # Health status: reliable, so downstream monitors never miss a warning
        status_qos = QoSProfile(
            reliability=ReliabilityPolicy.RELIABLE,
            durability=DurabilityPolicy.VOLATILE,
            history=HistoryPolicy.KEEP_LAST,
            depth=10,
        )

        # --- Subscriptions ---
        self._joint_state_sub = self.create_subscription(
            msg_type=JointState,
            topic='/joint_states',
            callback=self._joint_state_callback,
            qos_profile=sensor_qos,
        )

        # --- Publishers ---
        self._health_pub = self.create_publisher(
            msg_type=String,
            topic='/arm/health_status',
            qos_profile=status_qos,
        )

        # --- Internal state ---
        self._current_positions: JointPositionMap = {}
        self._message_count: int = 0
        self._warning_threshold_rad: float = math.radians(10.0)  # 10-degree warning zone

        # --- Diagnostic timer: log rate every 5 seconds ---
        self._rate_timer = self.create_timer(5.0, self._log_message_rate)
        self._last_rate_check_count: int = 0

        self.get_logger().info('JointStateMonitor initialized')

    def _joint_state_callback(self, msg: JointState) -> None:
        """
        Primary callback for joint state messages.
        Called at whatever frequency the hardware or simulation publishes joint states
        (typically 100–500 Hz for a real robot, or up to 1000 Hz in simulation).
        """
        self._message_count += 1

        # Update our position map from the received message
        for name, position in zip(msg.name, msg.position):
            self._current_positions[name] = position

        # Check for joints approaching limits
        warnings = self._check_position_limits()
        if warnings:
            status_msg = String()
            status_msg.data = (
                f'{{"timestamp": "{self.get_clock().now().nanoseconds}", '
                f'"warnings": {warnings}}}'
            )
            self._health_pub.publish(status_msg)

    def _check_position_limits(self) -> list[str]:
        """
        Checks all monitored joints against their limits.
        Returns a list of warning strings for joints within the warning threshold.
        """
        warnings: list[str] = []

        for joint_name, position in self._current_positions.items():
            if joint_name not in JOINT_LIMITS:
                continue  # Only check joints we have limits for

            lower_limit, upper_limit = JOINT_LIMITS[joint_name]

            # Check proximity to lower limit
            if position < lower_limit + self._warning_threshold_rad:
                margin_deg = math.degrees(position - lower_limit)
                warnings.append(
                    f'{joint_name}_near_lower_limit_{margin_deg:.1f}deg'
                )

            # Check proximity to upper limit
            elif position > upper_limit - self._warning_threshold_rad:
                margin_deg = math.degrees(upper_limit - position)
                warnings.append(
                    f'{joint_name}_near_upper_limit_{margin_deg:.1f}deg'
                )

        return warnings

    def _log_message_rate(self) -> None:
        """Timer callback: logs joint state message frequency every 5 seconds."""
        messages_in_period = self._message_count - self._last_rate_check_count
        rate_hz = messages_in_period / 5.0
        self._last_rate_check_count = self._message_count

        self.get_logger().info(
            f'Joint state receive rate: {rate_hz:.1f} Hz '
            f'(total messages: {self._message_count})'
        )

    @property
    def current_positions(self) -> JointPositionMap:
        """Read-only access to the latest joint positions."""
        return dict(self._current_positions)


def main(args: Optional[list[str]] = None) -> None:
    rclpy.init(args=args)
    node = JointStateMonitor()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## 4.3 Multi-Threaded Execution — Why It Matters for Real-Time Humanoid Control

The `rclpy.spin()` function used in Chapter 3's examples processes all callbacks in a single thread, one at a time. For a simple sensor logger, this is adequate. For a humanoid robot, it is a recipe for failure.

Consider a node that simultaneously:
- Subscribes to joint states at 500 Hz (every 2 ms)
- Provides a service to change control mode (occasionally, but must respond within 50 ms)
- Runs a 100 Hz control loop timer (every 10 ms)

With single-threaded execution, if the joint state callback takes 3 ms to process, the control loop timer's callback may miss its 10 ms deadline. Worse, a service call that arrives during a long perception computation will wait in the queue, potentially violating safety response time requirements.

ROS 2 solves this with **`MultiThreadedExecutor`** and **callback groups** [2].

### 4.3.1 Callback Groups

A **callback group** is an assignment of callbacks to either of two execution policies:

**`MutuallyExclusiveCallbackGroup`** — Only one callback in this group executes at a time. This is the default behavior and is safe for any callback that modifies shared state, since no two callbacks in the group can run concurrently.

**`ReentrantCallbackGroup`** — Multiple callbacks in this group can execute concurrently. This is appropriate for stateless callbacks (e.g., publishing a message based only on the received data), but requires the developer to protect shared state with locks.

### 4.3.2 Multi-Threaded Humanoid Control Node

```python
# multithreaded_humanoid_controller.py
# Demonstrates MultiThreadedExecutor and callback groups for a humanoid arm controller.
# ROS 2 Humble | Python 3.10+ | rclpy

from __future__ import annotations

import threading
from typing import Optional

import rclpy
from rclpy.node import Node
from rclpy.executors import MultiThreadedExecutor
from rclpy.callback_groups import MutuallyExclusiveCallbackGroup, ReentrantCallbackGroup
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy
from sensor_msgs.msg import JointState
from std_msgs.msg import Float64MultiArray
from std_srvs.srv import SetBool


class MultithreadedArmController(Node):
    """
    Humanoid arm controller demonstrating multi-threaded execution.

    Three callback groups:
    1. sensor_group (MutuallyExclusive): Processes joint state callbacks.
       Accesses self._current_positions — must not run concurrently with itself.
    2. control_group (MutuallyExclusive): Runs the 100 Hz control loop timer.
       Reads self._current_positions — must not run while joint state is updating.
    3. service_group (Reentrant): Handles control mode service requests.
       Stateless — can run concurrently with other service calls.

    By placing sensor_group and control_group in separate MutuallyExclusive groups,
    the joint state callback and control loop can run concurrently (in different threads),
    but two joint state callbacks will never overlap each other.
    """

    def __init__(self) -> None:
        super().__init__('multithreaded_arm_controller')

        # --- Callback groups ---
        # Separate group for sensor callbacks so they don't block the control loop
        self._sensor_group = MutuallyExclusiveCallbackGroup()
        # Separate group for the control loop timer
        self._control_group = MutuallyExclusiveCallbackGroup()
        # Reentrant group for service calls (stateless)
        self._service_group = ReentrantCallbackGroup()

        # --- Shared state protected by a lock ---
        self._state_lock = threading.Lock()
        self._current_positions: list[float] = [0.0, 0.0, 0.0]
        self._control_enabled: bool = False

        # --- QoS for joint states: high-frequency sensor data ---
        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST,
            depth=5,
        )

        # --- Subscription to joint states — uses sensor_group ---
        self._joint_state_sub = self.create_subscription(
            msg_type=JointState,
            topic='/joint_states',
            callback=self._joint_state_callback,
            qos_profile=sensor_qos,
            callback_group=self._sensor_group,
        )

        # --- Publisher for velocity commands ---
        self._velocity_pub = self.create_publisher(
            msg_type=Float64MultiArray,
            topic='/forward_velocity_controller/commands',
            qos_profile=10,
        )

        # --- Control loop timer at 100 Hz — uses control_group ---
        self._control_timer = self.create_timer(
            timer_period_sec=0.01,
            callback=self._control_loop,
            callback_group=self._control_group,
        )

        # --- Service to enable/disable control — uses service_group ---
        self._enable_service = self.create_service(
            srv_type=SetBool,
            srv_name='/arm_controller/enable',
            callback=self._handle_enable_request,
            callback_group=self._service_group,
        )

        self.get_logger().info(
            'MultithreadedArmController ready. '
            'POST to /arm_controller/enable to start.'
        )

    def _joint_state_callback(self, msg: JointState) -> None:
        """
        Runs in the sensor_group thread.
        Updates shared state under lock — safe for concurrent reads by control loop.
        """
        with self._state_lock:
            # Extract positions for our three arm joints
            for i, name in enumerate(msg.name):
                if name == 'shoulder_pitch' and len(msg.position) > i:
                    self._current_positions[0] = msg.position[i]
                elif name == 'elbow_flex' and len(msg.position) > i:
                    self._current_positions[1] = msg.position[i]
                elif name == 'wrist_roll' and len(msg.position) > i:
                    self._current_positions[2] = msg.position[i]

    def _control_loop(self) -> None:
        """
        Runs in the control_group thread at 100 Hz.
        Reads shared state under lock, computes velocity commands, publishes them.

        This callback runs CONCURRENTLY with _joint_state_callback because they are
        in DIFFERENT MutuallyExclusive groups. The lock ensures safe data access.
        """
        if not self._control_enabled:
            return

        # Read current state under lock
        with self._state_lock:
            positions = list(self._current_positions)

        # Simple proportional controller: drive joints toward zero (home position)
        target_positions = [0.0, 0.5, 0.0]  # elbow slightly bent
        kp = 2.0  # Proportional gain — rad/s per rad of error

        velocity_commands = [
            kp * (target - current)
            for target, current in zip(target_positions, positions)
        ]

        # Clamp to maximum velocity
        max_vel = 1.0  # rad/s
        velocity_commands = [
            max(-max_vel, min(max_vel, v)) for v in velocity_commands
        ]

        # Publish commands
        cmd_msg = Float64MultiArray()
        cmd_msg.data = velocity_commands
        self._velocity_pub.publish(cmd_msg)

    def _handle_enable_request(
        self,
        request: SetBool.Request,
        response: SetBool.Response,
    ) -> SetBool.Response:
        """
        Runs in the service_group thread (potentially reentrant).
        Enables or disables the control loop.
        """
        self._control_enabled = request.data
        action = 'enabled' if request.data else 'disabled'
        response.success = True
        response.message = f'Arm controller {action}'
        self.get_logger().info(response.message)
        return response


def main(args: Optional[list[str]] = None) -> None:
    rclpy.init(args=args)
    node = MultithreadedArmController()

    # MultiThreadedExecutor with 4 threads.
    # Each callback group's callbacks are dispatched to available threads.
    # The sensor_group and control_group can run simultaneously because they
    # have different group assignments and different threads are available.
    executor = MultiThreadedExecutor(num_threads=4)
    executor.add_node(node)

    try:
        executor.spin()
    except KeyboardInterrupt:
        pass
    finally:
        executor.shutdown()
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## 4.4 The ros2_control Framework

`ros2_control` [3] is the standard framework for hardware abstraction and deterministic control in ROS 2. It provides a clean separation between the **controller** (which implements control algorithms and is unaware of specific hardware) and the **hardware interface** (which implements hardware-specific read/write operations). This separation allows controllers to be developed and tested independently of the robot hardware.

### 4.4.1 Architecture Overview

The `ros2_control` architecture has three main components:

**Hardware Interface** — A C++ plugin that implements the `hardware_interface::SystemInterface` (or `SensorInterface`, `ActuatorInterface`) abstract class. It communicates with physical hardware via CAN bus, EtherCAT, serial, or any other protocol, and exposes the robot's state and command channels through `StateInterface` and `CommandInterface` objects.

**Controller Manager** — A ROS 2 node that loads hardware interface plugins and controller plugins, manages their lifecycle, and orchestrates the real-time control loop. The control loop runs at a configurable rate (typically 100–1000 Hz) and calls `hardware.read()`, then `controller.update()`, then `hardware.write()` in sequence.

**Controllers** — C++ or Python plugins that implement `controller_interface::ControllerInterface`. They read state from state interfaces and write commands to command interfaces. ROS 2 ships with a library of standard controllers: `JointTrajectoryController`, `ForwardCommandController`, `JointStateBroadcaster`, and others.

### 4.4.2 Hardware Interface Implementation

While hardware interfaces are typically written in C++ for maximum performance, the following Python pseudocode illustrates the conceptual structure. Production implementations should use the C++ API for real-time safety.

```python
# hardware_interface_concept.py
# Conceptual illustration of a ros2_control hardware interface.
# NOTE: Production hardware interfaces are written in C++ for real-time performance.
# This Python pseudocode illustrates the interface contract.
#
# The actual C++ base class is hardware_interface::SystemInterface from:
#   ros-humble-hardware-interface

from dataclasses import dataclass, field
from typing import Protocol


@dataclass
class StateInterface:
    """
    Exposes a read-only state value from hardware (e.g., current joint position).
    The Controller Manager makes these values available to all loaded controllers.
    """
    joint_name: str       # e.g., "shoulder_pitch"
    interface_name: str   # e.g., "position", "velocity", "effort"
    value: float = 0.0    # Current value, updated by hardware.read()


@dataclass
class CommandInterface:
    """
    Exposes a write-only command channel to hardware (e.g., target joint velocity).
    Controllers write to these; hardware.write() sends them to actuators.
    """
    joint_name: str
    interface_name: str   # e.g., "velocity", "position", "effort"
    value: float = 0.0    # Commanded value, written by the active controller


class HumanoidArmHardwareInterface:
    """
    Conceptual hardware interface for a 3-DOF humanoid arm.
    In production C++:
      class HumanoidArmHardware : public hardware_interface::SystemInterface { ... }

    The three lifecycle methods correspond to ros2_control lifecycle transitions:
      on_configure()  — open hardware connection, allocate buffers
      on_activate()   — enable servos, start real-time loop
      on_deactivate() — disable servos safely
    """

    JOINT_NAMES = ['shoulder_pitch', 'elbow_flex', 'wrist_roll']

    def __init__(self) -> None:
        # State interfaces: position and velocity for each joint
        self.state_interfaces: list[StateInterface] = [
            StateInterface(joint, interface_type)
            for joint in self.JOINT_NAMES
            for interface_type in ['position', 'velocity', 'effort']
        ]

        # Command interfaces: velocity commands for each joint
        self.command_interfaces: list[CommandInterface] = [
            CommandInterface(joint, 'velocity')
            for joint in self.JOINT_NAMES
        ]

    def read(self, time: float, period: float) -> None:
        """
        Called by Controller Manager at the start of each control cycle.
        Read current joint states from hardware and update state_interfaces.
        """
        for state_if in self.state_interfaces:

    def write(self, time: float, period: float) -> None:

        Called by Controller Manager at the end of each control cycle.
        Send command_interfaces values to hardware actuators.
        """

            # In production: write to motor driver register via CAN/EtherCAT
```

## 4.5 URDF Authoring for a Simplified 3-DOF Humanoid Arm

The **Unified Robot Description Format (URDF)** [4] is an XML specification for describing robot structure: links (rigid bodies), joints (connections between links), and their physical properties (mass, inertia, geometry, and collision bounds). ROS 2 tools — including RViz2, robot_state_publisher, and MoveIt 2 — consume URDF files to understand the robot's kinematic and dynamic structure.

The following is a complete, syntactically valid URDF for a 3-DOF humanoid arm. It includes visual geometry, collision geometry, inertial properties, and joint limits. Link dimensions are illustrative; a real system would derive them from CAD measurements.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!--
  humanoid_arm.urdf
  3-DOF humanoid arm: shoulder_pitch -> upper_arm -> elbow_flex -> forearm -> wrist_roll -> hand
  All lengths in meters, masses in kilograms, inertias in kg·m².
  Joint angles in radians.

  Kinematic chain:
    world -> base_link -> upper_arm_link [shoulder_pitch] ->
    forearm_link [elbow_flex] -> hand_link [wrist_roll]
-->
<robot name="humanoid_arm">

  <!-- ============================================================ -->
  <!-- LINKS                                                          -->
  <!-- ============================================================ -->

  <!-- Base link: fixed reference frame (torso attachment point) -->
  <link name="base_link">
    <inertial>
      <origin xyz="0 0 0" rpy="0 0 0"/>
      <mass value="2.0"/>
      <inertia ixx="0.01" ixy="0" ixz="0"
               iyy="0.01" iyz="0"
               izz="0.01"/>
    </inertial>
    <visual>
      <origin xyz="0 0 0.05" rpy="0 0 0"/>
      <geometry>
        <cylinder radius="0.05" length="0.10"/>
      </geometry>
      <material name="dark_grey">
        <color rgba="0.3 0.3 0.3 1.0"/>
      </material>
    </visual>
    <collision>
      <origin xyz="0 0 0.05" rpy="0 0 0"/>
      <geometry>
        <cylinder radius="0.05" length="0.10"/>
      </geometry>
    </collision>
  </link>

  <!-- Upper arm link: from shoulder to elbow -->
  <!-- Length: 0.30 m (approximately upper arm of a human adult) -->
  <link name="upper_arm_link">
    <inertial>
      <!-- Center of mass at geometric center of the cylinder -->
      <origin xyz="0 0 -0.15" rpy="0 0 0"/>
      <mass value="1.5"/>
      <!-- Solid cylinder: Ixx = Iyy = m(3r² + h²)/12, Izz = mr²/2 -->
      <!-- m=1.5 kg, r=0.04 m, h=0.30 m -->
      <inertia ixx="0.01184" ixy="0" ixz="0"
               iyy="0.01184" iyz="0"
               izz="0.00120"/>
    </inertial>
    <visual>
      <origin xyz="0 0 -0.15" rpy="0 0 0"/>
      <geometry>
        <cylinder radius="0.04" length="0.30"/>
      </geometry>
      <material name="light_grey">
        <color rgba="0.7 0.7 0.7 1.0"/>
      </material>
    </visual>
    <collision>
      <origin xyz="0 0 -0.15" rpy="0 0 0"/>
      <geometry>
        <!-- Slightly oversized collision cylinder for conservative planning -->
        <cylinder radius="0.045" length="0.32"/>
      </geometry>
    </collision>
  </link>

  <!-- Forearm link: from elbow to wrist -->
  <link name="forearm_link">
    <inertial>
```

## 4.6 MoveIt 2 Integration

**MoveIt 2** [5] is the standard motion planning framework for ROS 2. It provides:
- Inverse kinematics (IK) solvers (KDL, TRAC-IK, BioIK)
- Collision detection and avoidance (FCL — Flexible Collision Library)
- Motion planners (OMPL — Open Motion Planning Library, CHOMP, PILZ Industrial)
- A Python and C++ API for specifying goals in joint space or Cartesian space

### 4.6.1 MoveGroupInterface in Python

```python
# moveit2_arm_planner.py
# Plans and executes motions for the 3-DOF humanoid arm using MoveIt 2.
# ROS 2 Humble | Python 3.10+ | moveit_py
#
# Installation:
#   sudo apt install ros-humble-moveit ros-humble-moveit-py
#
# Note: moveit_py is the Python bindings for MoveIt 2 introduced in ROS 2 Humble.
# The older MoveGroupInterface (C++) is also available via the moveit_commander
# package, but moveit_py is the recommended Python interface for Humble.

from __future__ import annotations

import math
from typing import Optional

import rclpy
from rclpy.node import Node

# moveit_py provides Python bindings to MoveIt 2's core planning API
from moveit.planning import MoveItPy
from moveit.core.robot_state import RobotState


class HumanoidArmMoveItClient(Node):
    """
    Uses MoveIt 2 to plan and execute motions for the 3-DOF humanoid arm.

    Prerequisites:
    - MoveIt 2 configuration package for humanoid_arm must be running
      (generated by the MoveIt Setup Assistant)
    - move_group action server must be active
    - ros2_control controllers must be loaded and active
    """

    def __init__(self) -> None:
        super().__init__('humanoid_arm_moveit_client')

        # Initialize MoveItPy — connects to the running move_group node
        self._moveit = MoveItPy(node_name='moveit_py_client')

        # Get the planning component for the 'arm' planning group
        # Planning group name must match what was configured in Setup Assistant
        self._arm = self._moveit.get_planning_component('arm')

        self.get_logger().info('MoveIt 2 arm planning component ready')

    def plan_and_execute_joint_goal(
        self,
        joint_positions: dict[str, float],
        velocity_scaling: float = 0.3,
        acceleration_scaling: float = 0.1,
    ) -> bool:
        """
        Plans and executes a joint space goal.

        Args:
            joint_positions: Dict mapping joint name to target position in radians.
            velocity_scaling: Fraction of maximum velocity to use (0.0–1.0).
            acceleration_scaling: Fraction of maximum acceleration to use (0.0–1.0).

        Returns:
            True if planning and execution succeeded.
        """
        self.get_logger().info(f'Planning joint space goal: {joint_positions}')

        # Set the start state to the current robot state
        self._arm.set_start_state_to_current_state()

        # Create a goal state from the joint positions dictionary
        goal_state = self._build_joint_state(joint_positions)
        self._arm.set_goal_state(robot_state=goal_state)

        # Plan the motion
        # PlanRequestParameters allows setting planner ID and time limits
        from moveit.planning import PlanRequestParameters
        plan_params = PlanRequestParameters(
            self._moveit,
            'ompl_rrtconnect',  # OMPL RRT-Connect planner
        )
        plan_params.planning_time = 5.0       # seconds
        plan_params.max_velocity_scaling_factor = velocity_scaling
        plan_params.max_acceleration_scaling_factor = acceleration_scaling

        plan_result = self._arm.plan(single_plan_parameters=plan_params)

        if not plan_result:
            self.get_logger().error('Planning failed — goal may be unreachable or in collision')
            return False

        self.get_logger().info('Plan succeeded — executing...')

        # Execute the plan
        execute_result = self._moveit.execute(
            plan_result.trajectory,
            controllers=['joint_trajectory_controller'],
        )

        if execute_result:
            self.get_logger().info('Execution completed successfully')
            return True
        else:
            self.get_logger().error('Execution failed')
            return False

    def plan_cartesian_path(
        self,
        waypoints_xyz: list[tuple[float, float, float]],
        eef_step: float = 0.01,
    ) -> bool:
        """
        Plans a Cartesian path through a series of end-effector waypoints.
        The path follows straight lines between waypoints in Cartesian space.

        Args:
            waypoints_xyz: List of (x, y, z) end-effector positions in the
                           base_link frame (meters).
            eef_step: Maximum distance between consecutive Cartesian waypoints
                      for interpolation (meters). Smaller values = smoother path.

        Returns:
            True if a valid path covering all waypoints was planned and executed.
        """
        from geometry_msgs.msg import Pose
        import copy

        self.get_logger().info(
            f'Planning Cartesian path through {len(waypoints_xyz)} waypoints'
        )

        # Build a list of Pose messages from (x, y, z) tuples
        # Orientation is kept fixed (pointing downward — suitable for grasp approach)
        poses = []
        for x, y, z in waypoints_xyz:
            pose = Pose()
            pose.position.x = x
            pose.position.y = y
            pose.position.z = z
            # Fixed orientation: end-effector pointing straight down
            pose.orientation.x = 0.0
            pose.orientation.y = 1.0
            pose.orientation.z = 0.0
            pose.orientation.w = 0.0
            poses.append(pose)

        # Compute Cartesian path via MoveIt's compute_cartesian_path
        # fraction = fraction of path that is feasible (1.0 = fully feasible)
        self._arm.set_start_state_to_current_state()

        plan_result, fraction = self._arm.plan_cartesian_path(
            waypoints=poses,
            max_step=eef_step,
            jump_threshold=0.0,  # 0.0 disables jump checking (enable for safety in production)
        )

        self.get_logger().info(f'Cartesian path planned: {fraction * 100:.1f}% feasible')

        if fraction < 0.95:
            self.get_logger().warn(
                f'Only {fraction * 100:.1f}% of Cartesian path is feasible. '
                'Consider adjusting waypoints or checking for collisions.'
            )
            return False

        execute_result = self._moveit.execute(
            plan_result.trajectory,
            controllers=['joint_trajectory_controller'],
        )

        return bool(execute_result)

    def _build_joint_state(self, joint_positions: dict[str, float]) -> RobotState:
        """Creates a RobotState with the specified joint positions."""
        robot_model = self._moveit.get_robot_model()
        robot_state = RobotState(robot_model)
        robot_state.set_to_default_values()

        for joint_name, position in joint_positions.items():
            robot_state.set_joint_positions(joint_name, [position])

        robot_state.update()
        return robot_state


def main(args: Optional[list[str]] = None) -> None:
    rclpy.init(args=args)
    node = HumanoidArmMoveItClient()

    # Example 1: Plan and execute a joint space goal
    # Move to: shoulder_pitch = 0.5 rad, elbow_flex = 1.2 rad, wrist_roll = 0.0 rad
    success = node.plan_and_execute_joint_goal(
        joint_positions={
            'shoulder_pitch': 0.5,
            'elbow_flex': 1.2,
            'wrist_roll': 0.0,
        },
        velocity_scaling=0.3,
    )
    node.get_logger().info(f'Joint space goal: {"success" if success else "failed"}')

    # Example 2: Plan a Cartesian path (grasp approach trajectory)
    # Move end-effector through three waypoints above a table surface
    cartesian_success = node.plan_cartesian_path(
        waypoints_xyz=[
            (0.30, 0.0, 0.40),   # Approach point
            (0.30, 0.0, 0.25),   # Pre-grasp position
            (0.30, 0.0, 0.20),   # Grasp position
        ],
        eef_step=0.01,
    )
    node.get_logger().info(
        f'Cartesian path: {"success" if cartesian_success else "failed"}'
    )

    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## 4.7 Nav2 for Bipedal Navigation

**Nav2** (Navigation 2) [6] is the navigation stack for ROS 2. It provides autonomous navigation for mobile robots: path planning (global), local obstacle avoidance (local), behavior trees for task sequencing, and recovery behaviors. Nav2 is designed primarily for ground-contact wheeled robots, and adapting it to bipedal humanoids requires awareness of its assumptions and limitations.

### 4.7.1 Nav2 Architecture Overview

Nav2 consists of several major components:

**BT Navigator** — The top-level behavior tree executor that sequences the navigation task.

**Planner Server** — Computes a global path from the robot's current position to the goal using planners such as NavFn (Dijkstra-based), Smac Planner, and Theta\*.

**Controller Server** — Executes the global path by computing local velocity commands using DWB (Dynamic Window Approach), TEB (Timed Elastic Band), or MPPI (Model Predictive Path Integral) controllers.

**Costmap 2D** — Maintains a 2D occupancy grid of the environment with inflation layers and obstacle layers derived from laser scans or depth images.

**Recovery Server** — Handles stuck-robot situations with behaviors like spin, back-up, and clear costmap.

> **Figure 4.3 — Nav2 Architecture for a Wheeled Robot**
> *[Original diagram: Flow diagram showing BT Navigator at top calling Planner Server (global path), Controller Server (velocity commands), and Recovery Server. Show Costmap 2D feeding both servers. Show cmd_vel output going to robot base controller. Annotate each component with its ROS 2 action/topic interface. To be created in draw.io.]*

### 4.7.2 Caveats for Bipedal Humanoids

Nav2's default assumptions break down in important ways for bipedal humanoids:

**Footprint model.** Nav2 models the robot's footprint as a 2D polygon on the ground plane. A humanoid robot's stable footprint changes dramatically during walking — it is a point during single-support phase and a convex hull of both feet during double-support. Nav2 cannot model this dynamic footprint without custom modifications.

**Velocity commands.** Nav2 outputs `geometry_msgs/Twist` messages on `/cmd_vel` with linear x/y velocities and angular z velocity. A wheeled robot's motion controller maps these directly to wheel speeds. A bipedal humanoid's gait controller (based on ZMP, LIPM, or a reinforcement learning policy) receives much more complex inputs than `cmd_vel`. A translation layer — sometimes called a "gait adapter" — is required between Nav2's velocity commands and the humanoid's walking controller.

**Stability awareness.** Nav2 plans paths assuming the robot can follow any path that does not collide with obstacles. A biped may be unable to follow a curved path at the same speed as a straight path, or may be destabilized by sudden direction changes. A production humanoid navigation system would need to incorporate stability constraints into the planning layer.

**Research solutions.** Academic work on humanoid navigation with Nav2 — such as the Honda ASIMO extensions and work using the PAL Robotics TALOS platform [12] — typically replaces the controller server's velocity interface with a bipedal gait command interface and adds a stability costmap layer. These are active research areas as of 2026.

### 4.7.3 BehaviorTree XML Example

Nav2 uses **BehaviorTree.CPP** [7] for task sequencing. The following is a simplified BehaviorTree XML that illustrates a navigate-to-pose mission with a single recovery behavior:

```xml
<!--
  navigate_with_recovery.xml
  Simplified Nav2 behavior tree for a humanoid navigation task.
  Nav2 Humble | BehaviorTree.CPP v3

  Tree structure:
    RecoveryNode attempts NavigateToPose.
    If NavigateToPose fails, it spins in place and retries once.
-->
<root main_tree_to_execute="MainTree">
  <BehaviorTree ID="MainTree">

    <!-- RecoveryNode: tries the main action, executes recovery on failure -->
    <RecoveryNode number_of_retries="1" name="NavigateRecovery">

      <!-- Main action: plan and execute path to goal pose -->
      <PipelineSequence name="NavigateWithReplanning">

        <!-- Check if a valid global path exists; if not, replan -->
        <RateController hz="1.0">
          <ComputePathToPose goal="{goal}"
                             path="{path}"
                             planner_id="SmacPlanner2D"/>
        </RateController>

        <!-- Follow the computed path using the local controller -->
        <FollowPath path="{path}"
                    controller_id="FollowPath"
                    goal_checker_id="goal_checker"/>

      </PipelineSequence>

      <!-- Recovery subtree: executed when NavigateWithReplanning fails -->
      <ReactiveFallback name="RecoveryFallback">
        <!-- Check if the goal is still valid after recovery -->
        <GoalUpdated/>

        <!-- Recovery sequence: clear costmaps, then spin in place -->
        <SequenceStar name="RecoveryActions">
          <ClearEntireCostmap name="ClearLocalCostmap"
                              service_name="local_costmap/clear_entirely_local_costmap"/>
          <Spin spin_dist="1.57"/>  <!-- Spin 90 degrees -->
          <Wait wait_duration="1"/> <!-- Wait 1 second for sensors to update -->
        </SequenceStar>

      </ReactiveFallback>

    </RecoveryNode>

  </BehaviorTree>
</root>
```

## 4.8 Complete Example — Joint State Subscriber with Velocity Command Publisher

This section presents a self-contained, complete ROS 2 node that integrates the patterns from this chapter: subscribing to joint states, implementing a simple proportional controller, and publishing velocity commands via the `ros2_control` forward velocity controller interface.

```python
# arm_position_controller_node.py
# Complete example: reads joint states, runs P-controller, publishes velocity commands.
# ROS 2 Humble | Python 3.10+ | rclpy
#
# This node can be tested with the ros2_control mock hardware interface:
#   ros2 run controller_manager ros2_control_node --ros-args
#       -p robot_description:="$(cat humanoid_arm.urdf)"
#
# Then load controllers:
#   ros2 control load_controller joint_state_broadcaster
#   ros2 control load_controller forward_velocity_controller
#   ros2 control set_controller_state joint_state_broadcaster active
#   ros2 control set_controller_state forward_velocity_controller active

from __future__ import annotations

import threading
from typing import Optional

import rclpy
from rclpy.node import Node
from rclpy.executors import MultiThreadedExecutor
from rclpy.callback_groups import MutuallyExclusiveCallbackGroup
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy
from sensor_msgs.msg import JointState
from std_msgs.msg import Float64MultiArray

# Target joint positions (radians) — a relaxed arm pose
TARGET_POSITIONS: dict[str, float] = {
    'shoulder_pitch': 0.3,    # 17 degrees forward
    'elbow_flex':     0.8,    # ~46 degrees bent
    'wrist_roll':     0.0,    # Neutral wrist rotation
}

# Proportional controller gains (rad/s per rad of position error)
KP: dict[str, float] = {
    'shoulder_pitch': 1.5,
    'elbow_flex':     1.5,
    'wrist_roll':     2.0,
}

# Velocity limits (rad/s) — must not exceed URDF joint velocity limits
MAX_VELOCITY: dict[str, float] = {
    'shoulder_pitch': 1.0,
    'elbow_flex':     1.0,
    'wrist_roll':     2.0,
}

JOINT_ORDER = ['shoulder_pitch', 'elbow_flex', 'wrist_roll']


class ArmPositionControllerNode(Node):
    """
    Complete arm position controller node.

    Subscribes to /joint_states (from JointStateBroadcaster).
    Runs a proportional position controller at 100 Hz.
    Publishes velocity commands to /forward_velocity_controller/commands.

    Uses MultiThreadedExecutor so the 100 Hz control loop is not blocked
    by the joint state callback processing.
    """

    def __init__(self) -> None:
        super().__init__('arm_position_controller')

        # --- Callback groups for concurrent execution ---
        self._sensor_cbg = MutuallyExclusiveCallbackGroup()
        self._control_cbg = MutuallyExclusiveCallbackGroup()

        # --- Shared state ---
        self._state_lock = threading.Lock()
        self._current_positions: dict[str, float] = {
            name: 0.0 for name in JOINT_ORDER
        }
        self._joint_states_received: bool = False

        # --- Subscription ---
        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST,
            depth=5,
        )
        self._joint_state_sub = self.create_subscription(
            msg_type=JointState,
            topic='/joint_states',
            callback=self._joint_state_callback,
            qos_profile=sensor_qos,
            callback_group=self._sensor_cbg,
        )

        # --- Publisher to forward velocity controller ---
        self._velocity_pub = self.create_publisher(
            msg_type=Float64MultiArray,
            topic='/forward_velocity_controller/commands',
            qos_profile=10,
        )

        # --- Control loop at 100 Hz ---
        self._control_timer = self.create_timer(
            timer_period_sec=0.01,
            callback=self._control_loop,
            callback_group=self._control_cbg,
        )

        # --- Diagnostic timer at 1 Hz ---
        self._diag_timer = self.create_timer(
            timer_period_sec=1.0,
            callback=self._log_state,
        )

        self.get_logger().info(
            'ArmPositionControllerNode started. '
            'Waiting for joint states on /joint_states...'
        )

    def _joint_state_callback(self, msg: JointState) -> None:
        """
        Updates current joint positions from the JointStateBroadcaster.
        Uses lock to protect shared state from concurrent control loop access.
        """
        with self._state_lock:
            for name, position in zip(msg.name, msg.position):
                if name in self._current_positions:
                    self._current_positions[name] = position
            self._joint_states_received = True

    def _control_loop(self) -> None:
        """
        Proportional position controller running at 100 Hz.
        Reads current joint positions, computes velocity commands, publishes.
        """
        with self._state_lock:
            if not self._joint_states_received:
                return  # Do not command until we have state feedback
            current = dict(self._current_positions)

        velocity_commands: list[float] = []

        for joint_name in JOINT_ORDER:
            error = TARGET_POSITIONS[joint_name] - current[joint_name]
            raw_command = KP[joint_name] * error

            # Apply velocity clamping for safety
            max_vel = MAX_VELOCITY[joint_name]
            clamped_command = max(-max_vel, min(max_vel, raw_command))
            velocity_commands.append(clamped_command)

        # Publish to the forward velocity controller
        cmd_msg = Float64MultiArray()
        cmd_msg.data = velocity_commands
        self._velocity_pub.publish(cmd_msg)

    def _log_state(self) -> None:
        """Logs current joint positions and velocity errors at 1 Hz."""
        with self._state_lock:
            current = dict(self._current_positions)

        for joint_name in JOINT_ORDER:
            error_deg = (TARGET_POSITIONS[joint_name] - current[joint_name])
            error_deg_display = error_deg * (180.0 / 3.14159)
            self.get_logger().info(
                f'{joint_name}: '
                f'current={current[joint_name]:.3f} rad, '
                f'target={TARGET_POSITIONS[joint_name]:.3f} rad, '
                f'error={error_deg_display:.1f} deg'
            )


def main(args: Optional[list[str]] = None) -> None:
    rclpy.init(args=args)
    node = ArmPositionControllerNode()

    executor = MultiThreadedExecutor(num_threads=3)
    executor.add_node(node)

    try:
        executor.spin()
    except KeyboardInterrupt:
        node.get_logger().info('Shutting down arm controller')
    finally:
        # Publish zero velocities before shutting down — safety measure
        zero_msg = Float64MultiArray()
        zero_msg.data = [0.0, 0.0, 0.0]
        node._velocity_pub.publish(zero_msg)

        executor.shutdown()
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

This chapter translated ROS 2 architecture into practical humanoid robot control software.

We developed production-quality Python nodes with type annotations and proper QoS configuration, establishing the structural patterns that make ROS 2 code maintainable at scale. The chapter closed with a complete, self-contained proportional position controller that synthesizes multi-threading, joint state subscription, and velocity command publishing into a runnable, testable node.
