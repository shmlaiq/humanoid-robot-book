## 3.1 Introduction — Why Robots Need Middleware

Imagine building a humanoid robot without any shared communication infrastructure. Every sensor driver, every motion controller, and every perception algorithm would need to speak directly to every other component using bespoke, hand-crafted protocols. A LiDAR driver would need to know the exact memory layout expected by the SLAM algorithm. The joint velocity controller would need to directly poll the IMU at precisely the right moment. Adding a new camera would require modifying every module that consumes visual data. The result would be a tightly coupled monolith — fragile, untestable, and impossible to port between robot platforms.

This is precisely the problem that **middleware** solves. In computing, middleware sits between application software and the underlying operating system, providing a standardized set of communication, discovery, and data management services. For robotics, the challenge is uniquely demanding: middleware must handle real-time sensor streams, coordinate dozens of concurrent processes running across multiple CPU cores (or even multiple networked computers), and do all of this with latency measured in milliseconds.

**ROS 2 (Robot Operating System 2)** [1] is the de facto standard robotic middleware as of 2026. It is not a traditional operating system — it runs atop Linux (or Windows or macOS) and provides a rich set of libraries, tools, and conventions that allow software developers to build robot applications as loosely coupled, independently testable components called **nodes**. Nodes communicate through well-defined interfaces: topics for streaming data, services for request-response transactions, and actions for long-running tasks with feedback. The entire communication layer is built on top of the **DDS (Data Distribution Service)** standard [2], a publish-subscribe middleware specification maintained by the Object Management Group (OMG) that brings with it industrial-grade reliability, quality-of-service controls, and optional security.

For a humanoid robot, this architecture is not a convenience — it is a necessity. A system with 28–44 degrees of freedom, multiple cameras, IMUs, force-torque sensors, microphones, and an onboard GPU running inference workloads simultaneously cannot be coordinated by a monolithic program. It demands an architecture where each concern is isolated in its own node, where failures are contained rather than propagating through the entire system, and where new capabilities can be added by writing new nodes without touching existing ones.

This chapter provides a complete technical grounding in ROS 2 architecture — from the history of why it replaced its predecessor, through the DDS middleware layer, the four core communication primitives, lifecycle node management, and the build system, to a full workspace setup guide. Every concept is illustrated with code that runs on a standard Ubuntu 22.04 development machine.

## 3.2 Why ROS 2 Replaced ROS 1

ROS 1, first released by Willow Garage in 2007 [3], was a transformative achievement. It gave the robotics research community a common software platform and enabled an explosion of open-source robot algorithms. However, ROS 1 was designed with a set of architectural assumptions that proved inadequate for production deployment:

**Single master dependency.** ROS 1 required a central process called `rosmaster` to broker all communication. If this process crashed, the entire robot communication infrastructure failed — an unacceptable single point of failure for a deployed system.

**No real-time support.** ROS 1 was designed for research robots running on standard Linux kernels. It provided no latency guarantees, making it unsuitable for hard real-time control loops that require deterministic execution within microseconds.

**No security.** ROS 1 nodes communicated over unencrypted TCP/UDP connections with no authentication. On a research network, this was acceptable. On a humanoid robot operating in a hospital or factory, it is a critical vulnerability — any device on the same network can inject arbitrary commands.

**Multi-robot limitations.** Coordinating multiple robots in ROS 1 required complex namespacing conventions and was never a first-class design concern.

**Python 2 dependency.** ROS 1 was tightly coupled to Python 2, which reached end-of-life in January 2020.

ROS 2, developed by Open Robotics (now part of the Intrinsic organization at Alphabet) in collaboration with industry partners, was redesigned from the ground up to address all of these limitations [1]. Its first stable release was Ardent Apalone in 2017. The current Long Term Support (LTS) release is **ROS 2 Humble Hawksbill**, released in May 2022 and supported through May 2027 on **Ubuntu 22.04 LTS** [4].

The four architectural advances that define ROS 2:

### 3.2.1 DDS as the Communication Foundation

Rather than building a custom transport layer, ROS 2 delegates all communication to vendor implementations of the **DDS (Data Distribution Service)** standard [2]. DDS is a peer-to-peer publish-subscribe middleware that was originally developed for defense, aerospace, and medical systems — exactly the domains that demand reliability and real-time performance. By using DDS, ROS 2 inherits decades of industrial middleware engineering and gains the ability to plug in different DDS vendors depending on use case requirements.

### 3.2.2 Real-Time Support

ROS 2 is designed to work with real-time Linux kernels (PREEMPT-RT patches) and provides mechanisms for deterministic execution. The `ros2_control` framework (Chapter 4) operates control loops with deterministic timing on dedicated threads. Quality-of-service (QoS) policies allow developers to specify reliability and deadline requirements for each communication channel.

### 3.2.3 Security via DDS-Security

ROS 2 integrates with the **DDS-Security** specification [5] through the SROS2 (Secure ROS 2) tooling, providing mutual authentication, access control, and encryption for all inter-node communication. This is covered in depth in Chapter 5.

### 3.2.4 Multi-Robot and Multi-Host Support

Because DDS is fundamentally a networked, peer-to-peer protocol, multiple robots can participate in the same ROS 2 graph simply by being on the same network and sharing a **Domain ID**. Isolation between robot systems is achieved by using different Domain IDs. No central master process is required.

## 3.3 DDS Middleware Deep Dive

Understanding DDS is essential for diagnosing performance issues, configuring quality-of-service behavior, and understanding the security model. DDS operates on a publish-subscribe paradigm: **publishers** write data to named **topics**, and **subscribers** receive data from those topics. The DDS layer handles discovery, matching publishers to subscribers, and delivering data according to configurable quality policies.

### 3.3.1 The RTPS Wire Protocol

DDS implementations communicate using the **RTPS (Real-Time Publish-Subscribe)** protocol [2], which defines the wire format for discovery messages and data exchange. RTPS runs over UDP/IP (by default) or TCP/IP and supports both unicast and multicast communication. When a ROS 2 node starts, it uses RTPS's **Simple Discovery Protocol (SDP)** to announce its publishers and subscribers on the network, allowing other nodes to discover and connect to it without a central broker.

### 3.3.2 DDS Vendor Implementations in ROS 2

ROS 2 supports multiple DDS vendor implementations through an abstraction layer called the **RMW (ROS 2 Middleware Interface)**. The two most widely used implementations are:

**Fast DDS (eProsima)** — The default DDS implementation for ROS 2 Humble. Fast DDS is open source under the Apache 2.0 license and is maintained by eProsima. It supports all DDS-Security features and provides good performance for most robotics workloads. The RMW package is `rmw_fastrtps_cpp`.

**Cyclone DDS (Eclipse Foundation)** — An alternative open-source DDS implementation, originally developed at ADLINK Technology and now maintained by the Eclipse Foundation. Cyclone DDS is known for lower latency in some configurations and is widely used in production robotics deployments. The RMW package is `rmw_cyclonedds_cpp`.

Switching between DDS implementations requires only setting an environment variable:

```bash
# Use Cyclone DDS
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp

# Use Fast DDS (default for ROS 2 Humble)
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
```

### 3.3.3 Quality of Service Policies

One of the most powerful features DDS brings to ROS 2 is a rich set of **Quality of Service (QoS)** policies that allow developers to express requirements about how data should be delivered. The following table summarizes the QoS policies most relevant to robotics applications:

| QoS Policy | Options | Use Case |
|---|---|---|
| **Reliability** | RELIABLE, BEST_EFFORT | Use RELIABLE for sensor data that must not be dropped; BEST_EFFORT for high-frequency data where staleness is worse than loss |
| **Durability** | VOLATILE, TRANSIENT_LOCAL | Use TRANSIENT_LOCAL to deliver the last published message to late-joining subscribers (useful for robot description topics) |
| **History** | KEEP_LAST(N), KEEP_ALL | KEEP_LAST(1) for real-time sensor streams; KEEP_ALL for audit logs |
| **Deadline** | Duration | Publisher commits to publishing within the deadline; subscriber is notified if deadline is missed |
| **Liveliness** | AUTOMATIC, MANUAL_BY_PARTICIPANT | Detects when a publisher becomes unresponsive — critical for fault detection |
| **Lifespan** | Duration | Data older than the lifespan is discarded — prevents stale sensor readings from being consumed |

ROS 2 provides several pre-configured QoS profiles for common use cases. The `SENSOR_DATA` profile uses BEST_EFFORT reliability and KEEP_LAST(5) history — appropriate for camera images where a dropped frame is preferable to a delayed one. The `SERVICES_DEFAULT` profile uses RELIABLE reliability — appropriate for service calls that must complete successfully.

### 3.3.4 Discovery and Domain IDs

Every ROS 2 process is assigned to a **Domain ID** (an integer from 0 to 232). Nodes in different Domain IDs cannot discover each other, providing a simple isolation mechanism. The Domain ID is set via the `ROS_DOMAIN_ID` environment variable:

```bash
# Isolate this terminal session to Domain 42
export ROS_DOMAIN_ID=42
```

Within a domain, DDS discovery uses multicast UDP (on local networks) or configured peer lists (for cross-subnet communication) to allow nodes to find each other automatically. This peer-to-peer discovery eliminates the single point of failure that plagued ROS 1's `rosmaster`.

## 3.4 Core Communication Primitives

ROS 2 provides four communication primitives, each designed for a different interaction pattern. Understanding when to use each primitive is fundamental to designing well-structured robot software.

### 3.4.1 Topics — Asynchronous Publish/Subscribe

**Topics** are the primary mechanism for streaming data in ROS 2. A publisher node writes messages to a named topic, and any number of subscriber nodes can receive those messages. Communication is asynchronous — the publisher does not wait for subscribers to process the message, and subscribers do not request data on demand. This makes topics ideal for continuous sensor streams such as camera images, LiDAR point clouds, joint states, and IMU readings.

Messages on topics are typed. Each topic has an associated message type (defined in `.msg` files), and the ROS 2 type system ensures that publishers and subscribers agree on the message format at connection time.

The following example demonstrates a publisher node that publishes joint velocity commands at 100 Hz, and a subscriber node that receives and logs them:

```python
# joint_velocity_publisher.py
# Publishes joint velocity commands for a 3-DOF humanoid arm at 100 Hz.
# ROS 2 Humble | Python 3.10+ | rclpy

import rclpy
from rclpy.node import Node
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy
from sensor_msgs.msg import JointState
import math


class JointVelocityPublisher(Node):
    """
    Publishes sinusoidal joint velocity commands on the /joint_velocity_commands
    topic. Intended to demonstrate topic-based communication in ROS 2.
    """

    def __init__(self) -> None:
        super().__init__('joint_velocity_publisher')

        # Define QoS profile: best-effort reliability for high-frequency control data.
        # Dropping an occasional message is preferable to queuing stale commands.
        qos_profile = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST,
            depth=10,
        )

        # Create the publisher on the /joint_velocity_commands topic
        self._publisher = self.create_publisher(
            msg_type=JointState,
            topic='/joint_velocity_commands',
            qos_profile=qos_profile,
        )

        # Timer fires at 100 Hz (period = 0.01 seconds)
        self._timer = self.create_timer(
            timer_period_sec=0.01,
            callback=self._publish_velocities,
        )

        # Track elapsed time for sinusoidal trajectory generation
        self._elapsed_time: float = 0.0

        self.get_logger().info('JointVelocityPublisher started at 100 Hz')

    def _publish_velocities(self) -> None:
        """Callback invoked by the timer every 10 ms. Publishes one JointState message."""
        msg = JointState()
        msg.header.stamp = self.get_clock().now().to_msg()

        # Three joints: shoulder_pitch, elbow_flex, wrist_roll
        msg.name = ['shoulder_pitch', 'elbow_flex', 'wrist_roll']

        # Sinusoidal velocity profile — amplitude 0.5 rad/s, frequency 0.5 Hz
        msg.velocity = [
            0.5 * math.sin(2.0 * math.pi * 0.5 * self._elapsed_time),
            0.3 * math.sin(2.0 * math.pi * 0.3 * self._elapsed_time + 0.5),
            0.2 * math.sin(2.0 * math.pi * 0.2 * self._elapsed_time + 1.0),
        ]

        self._publisher.publish(msg)
        self._elapsed_time += 0.01  # Advance by one timer period


def main(args=None) -> None:
    rclpy.init(args=args)
    node = JointVelocityPublisher()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

```python
# joint_velocity_subscriber.py
# Subscribes to joint velocity commands and logs received data.
# ROS 2 Humble | Python 3.10+ | rclpy

import rclpy
from rclpy.node import Node
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy
from sensor_msgs.msg import JointState


class JointVelocitySubscriber(Node):
    """
    Subscribes to the /joint_velocity_commands topic and logs each received message.
    Demonstrates how QoS profiles must be compatible between publisher and subscriber.
    """

    def __init__(self) -> None:
        super().__init__('joint_velocity_subscriber')

        # QoS profile must be compatible with the publisher's profile.
        # BEST_EFFORT subscriber can receive from BEST_EFFORT publisher.
        qos_profile = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST,
            depth=10,
        )

        self._subscription = self.create_subscription(
            msg_type=JointState,
            topic='/joint_velocity_commands',
            callback=self._velocity_callback,
            qos_profile=qos_profile,
        )

        self._message_count: int = 0

    def _velocity_callback(self, msg: JointState) -> None:
        """Called each time a new JointState message is received."""
        self._message_count += 1

        # Log every 100th message to avoid flooding the terminal
        if self._message_count % 100 == 0:
            vel_str = ', '.join(
                f'{name}: {vel:.3f} rad/s'
                for name, vel in zip(msg.name, msg.velocity)
            )
            self.get_logger().info(
                f'[msg #{self._message_count}] Velocities: {vel_str}'
            )


def main(args=None) -> None:
    rclpy.init(args=args)
    node = JointVelocitySubscriber()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### 3.4.2 Services — Synchronous Request/Response

**Services** implement a synchronous request-response pattern. A service server advertises a named service with a defined request and response type. A service client sends a request and blocks (or uses a callback) until the server returns a response. Services are appropriate for discrete operations that must complete before the caller can proceed — for example, querying the current kinematic configuration, setting a control mode, or triggering a calibration routine.

Services are not appropriate for continuous data streams, as each transaction requires a round-trip through the communication layer. Calling a service at control-loop frequency (100+ Hz) would impose unacceptable overhead.

```python
# gripper_service.py
# Demonstrates a service server that opens/closes a gripper on command.
# ROS 2 Humble | Python 3.10+ | rclpy
# Uses the standard SetBool service type: request.data=True opens, False closes.

import rclpy
from rclpy.node import Node
from std_srvs.srv import SetBool


class GripperControlServer(Node):
    """
    Service server that controls a binary gripper (open/closed).
    Service type: std_srvs/srv/SetBool
    Service name: /gripper/set_state
    """

    def __init__(self) -> None:
        super().__init__('gripper_control_server')

        # Create the service server
        self._service = self.create_service(
            srv_type=SetBool,
            srv_name='/gripper/set_state',
            callback=self._handle_gripper_request,
        )

        # Track current gripper state
        self._is_open: bool = False

        self.get_logger().info('GripperControlServer ready on /gripper/set_state')

    def _handle_gripper_request(
        self,
        request: SetBool.Request,
        response: SetBool.Response,
    ) -> SetBool.Response:
        """
        Handles an incoming SetBool request.
        request.data: True = open gripper, False = close gripper
        response.success: True if state changed or already in desired state
        response.message: Human-readable description of the outcome
        """
        desired_open: bool = request.data
        action_str = 'open' if desired_open else 'close'

        if self._is_open == desired_open:
            # Already in the desired state — no hardware command required
            response.success = True
            response.message = f'Gripper already {action_str}ed; no action taken'
        else:
            # In a real system, this would send a hardware command to the gripper.
            # Here we simulate successful execution.
            self._is_open = desired_open
            response.success = True
            response.message = f'Gripper successfully {action_str}ed'
            self.get_logger().info(response.message)

        return response


# --- Client side ---

class GripperControlClient(Node):
    """
    Service client that sends open/close commands to the GripperControlServer.
    Demonstrates both synchronous (spin_until_future_complete) and asynchronous usage.
    """

    def __init__(self) -> None:
        super().__init__('gripper_control_client')

        self._client = self.create_client(
            srv_type=SetBool,
            srv_name='/gripper/set_state',
        )

        # Wait until the server is available before proceeding
        while not self._client.wait_for_service(timeout_sec=1.0):
            self.get_logger().info('Waiting for /gripper/set_state service...')

    def send_command(self, open_gripper: bool) -> bool:
        """
        Sends a gripper command and waits for the response.
        Returns True if the command was accepted successfully.
        """
        request = SetBool.Request()
        request.data = open_gripper

        # send_request is asynchronous; spin_until_future_complete makes it synchronous
        future = self._client.call_async(request)
        rclpy.spin_until_future_complete(self, future)

        result: SetBool.Response = future.result()
        self.get_logger().info(f'Service response: {result.message}')
        return result.success
```

### 3.4.3 Actions — Long-Running Tasks with Feedback

**Actions** are ROS 2's mechanism for long-running tasks that produce intermediate feedback. An action server accepts a **goal** from an action client, executes it over time while publishing **feedback** messages, and returns a **result** when the task completes (or an error if it fails or is cancelled). The client can cancel the goal at any time.

Actions are essential for humanoid robot operations such as navigating to a waypoint, executing a manipulation sequence, or performing a calibration routine — tasks that take seconds to minutes and where the client needs progress updates. Internally, actions are implemented as a combination of three topics and two services, but the rclpy API presents a clean, unified abstraction.

```python
# joint_trajectory_action.py
# Action server that executes a joint trajectory for a 3-DOF arm.
# ROS 2 Humble | Python 3.10+ | rclpy
# Uses control_msgs/action/FollowJointTrajectory as the action type.
#
# NOTE: This example uses a simplified custom action to avoid external dependencies.
# In production, use control_msgs/action/FollowJointTrajectory from ros2_control.

import time
import rclpy
from rclpy.node import Node
from rclpy.action import ActionServer, ActionClient, CancelResponse, GoalResponse
from rclpy.action.server import ServerGoalHandle

# We use the standard FollowJointTrajectory action type from control_msgs.
# Ensure ros-humble-control-msgs is installed:
#   sudo apt install ros-humble-control-msgs
from control_msgs.action import FollowJointTrajectory
from trajectory_msgs.msg import JointTrajectoryPoint
import builtin_interfaces.msg


class JointTrajectoryActionServer(Node):
    """
    Action server that executes joint trajectories for a simulated 3-DOF arm.
    Publishes feedback (current joint positions) during execution.
    """

    def __init__(self) -> None:
        super().__init__('joint_trajectory_action_server')

        self._action_server = ActionServer(
            node=self,
            action_type=FollowJointTrajectory,
            action_name='/arm_controller/follow_joint_trajectory',
            execute_callback=self._execute_trajectory,
            goal_callback=self._goal_callback,
            cancel_callback=self._cancel_callback,
        )

        # Simulated current joint positions (radians)
        self._current_positions: list[float] = [0.0, 0.0, 0.0]

        self.get_logger().info('JointTrajectoryActionServer ready')

    def _goal_callback(self, goal_request: FollowJointTrajectory.Goal) -> GoalResponse:
        """Called when a new goal arrives. Returns ACCEPT or REJECT."""
        num_points = len(goal_request.trajectory.points)
        self.get_logger().info(f'Received trajectory goal with {num_points} waypoints')

        # Reject empty trajectories
        if num_points == 0:
            self.get_logger().warn('Rejecting empty trajectory')
            return GoalResponse.REJECT

        return GoalResponse.ACCEPT

    def _cancel_callback(self, goal_handle: ServerGoalHandle) -> CancelResponse:
        """Called when the client requests cancellation. Returns ACCEPT or REJECT."""
        self.get_logger().info('Received cancellation request — accepting')
        return CancelResponse.ACCEPT

    def _execute_trajectory(
        self, goal_handle: ServerGoalHandle
    ) -> FollowJointTrajectory.Result:
        """
        Main execution callback. Called in a separate thread when a goal is accepted.
        Iterates through trajectory waypoints, publishing feedback at each step.
        """
        trajectory = goal_handle.request.trajectory
        feedback_msg = FollowJointTrajectory.Feedback()

        self.get_logger().info(
            f'Executing trajectory with {len(trajectory.points)} waypoints'
        )

        for i, point in enumerate(trajectory.points):
            # Check if the client has requested cancellation
            if goal_handle.is_cancel_requested:
                goal_handle.canceled()
                self.get_logger().info('Trajectory execution cancelled')
                return FollowJointTrajectory.Result()

            # Simulate moving toward the target point
            self._current_positions = list(point.positions)

            # Publish feedback: current positions, desired positions, and error
            feedback_msg.joint_names = trajectory.joint_names
            feedback_msg.actual.positions = self._current_positions
            feedback_msg.desired.positions = list(point.positions)
            feedback_msg.error.positions = [0.0] * len(point.positions)

            goal_handle.publish_feedback(feedback_msg)

            self.get_logger().info(
                f'Waypoint {i + 1}/{len(trajectory.points)}: '
                f'positions = {[f"{p:.3f}" for p in self._current_positions]}'
            )

            # Simulate execution time — in a real system, poll hardware state
            time.sleep(0.1)

        # All waypoints executed successfully
        goal_handle.succeed()
        result = FollowJointTrajectory.Result()
        self.get_logger().info('Trajectory execution completed successfully')
        return result
```

### 3.4.4 Parameters — Runtime Configuration

**Parameters** allow ROS 2 nodes to expose configuration values that can be read and modified at runtime without restarting the node. Parameters are strongly typed (bool, int, double, string, arrays of each) and are declared by the node at startup. Other nodes can read and set parameters via a standard service interface, and parameter changes can trigger callbacks in the node.

For a humanoid robot, parameters are essential for values that need to be tuned without recompiling — PID controller gains, velocity limits, sensor topic names, and safety thresholds.

```python
# velocity_controller_with_params.py
# Demonstrates ROS 2 parameters for a joint velocity controller.
# ROS 2 Humble | Python 3.10+ | rclpy

import rclpy
from rclpy.node import Node
from rclpy.parameter import Parameter
from rcl_interfaces.msg import ParameterDescriptor, SetParametersResult


class VelocityControllerWithParams(Node):
    """
    A joint velocity controller that exposes its configuration as ROS 2 parameters.
    Parameters can be set from the command line:
        ros2 param set /velocity_controller max_velocity 1.5
    Or from a YAML parameter file passed at launch time.
    """

    def __init__(self) -> None:
        super().__init__('velocity_controller')

        # --- Declare parameters with types, defaults, and descriptions ---

        self.declare_parameter(
            name='max_velocity',
            value=1.0,  # rad/s
            descriptor=ParameterDescriptor(
                description='Maximum joint velocity in rad/s. Hard limit enforced.',
            ),
        )

        self.declare_parameter(
            name='control_frequency',
            value=100.0,  # Hz
            descriptor=ParameterDescriptor(
                description='Control loop update frequency in Hz.',
            ),
        )

        self.declare_parameter(
            name='joint_names',
            value=['shoulder_pitch', 'elbow_flex', 'wrist_roll'],
            descriptor=ParameterDescriptor(
                description='List of joint names this controller manages.',
            ),
        )

        self.declare_parameter(
            name='enable_safety_limits',
            value=True,
            descriptor=ParameterDescriptor(
                description='If True, velocity commands are clamped to max_velocity.',
            ),
        )

        # Register a callback invoked when any parameter changes at runtime
        self.add_on_set_parameters_callback(self._on_parameter_change)

        # Read initial parameter values into instance variables
        self._load_parameters()

        # Start the control loop timer using the declared frequency
        period = 1.0 / self._control_frequency
        self._timer = self.create_timer(period, self._control_loop)

        self.get_logger().info(
            f'VelocityController started: '
            f'frequency={self._control_frequency} Hz, '
            f'max_velocity={self._max_velocity} rad/s, '
            f'joints={self._joint_names}'
        )

    def _load_parameters(self) -> None:
        """Read current parameter values into instance variables."""
        self._max_velocity: float = (
            self.get_parameter('max_velocity').get_parameter_value().double_value
        )
        self._control_frequency: float = (
            self.get_parameter('control_frequency').get_parameter_value().double_value
        )
        self._joint_names: list[str] = (
            self.get_parameter('joint_names').get_parameter_value().string_array_value
        )
        self._enable_safety_limits: bool = (
            self.get_parameter('enable_safety_limits')
            .get_parameter_value()
            .bool_value
        )

    def _on_parameter_change(
        self, params: list[Parameter]
    ) -> SetParametersResult:
        """
        Callback invoked before a parameter change is committed.
        Validate new values and return success/failure.
        """
        for param in params:
            if param.name == 'max_velocity':
                if param.value <= 0.0 or param.value > 5.0:
                    return SetParametersResult(
                        successful=False,
                        reason='max_velocity must be in range (0.0, 5.0] rad/s',
                    )
            if param.name == 'control_frequency':
                if param.value <= 0.0 or param.value > 1000.0:
                    return SetParametersResult(
                        successful=False,
                        reason='control_frequency must be in range (0.0, 1000.0] Hz',
                    )

        # All validations passed — reload parameters into instance variables
        self._load_parameters()
        self.get_logger().info(f'Parameters updated: {[p.name for p in params]}')
        return SetParametersResult(successful=True)
```

## 3.5 Node Lifecycle Management

Standard ROS 2 nodes start executing immediately when created and continue until the process terminates. For production systems, this is insufficient. A humanoid robot powering on needs a deterministic, ordered startup sequence: sensors must be initialized before perception algorithms read from them, hardware interfaces must be verified before controllers command them, and the system must be able to transition to a safe, low-power state without a full shutdown.

**Lifecycle nodes** [6] (also called managed nodes) implement a finite state machine that gives external orchestrators fine-grained control over node execution. The four primary states are:

**Unconfigured** — The node has been constructed but not yet initialized. No resources (timers, publishers, subscribers, hardware connections) have been allocated.

**Inactive** — The node has been configured (resources allocated, parameters loaded, hardware connections verified) but is not yet processing data or publishing commands. This is a "ready but paused" state.

**Active** — The node is fully operational: processing data, publishing outputs, executing control loops.

**Finalized** — The node has been cleanly shut down. All resources have been released.

Transitions between states are triggered by external commands (`configure`, `activate`, `deactivate`, `cleanup`, `shutdown`) and can be automated by a launch system or a system health manager node.

Show error states and error handling transitions. Arrows should be labeled with both the transition command and the callback name. To be created in draw.io.]*

```python
# imu_lifecycle_node.py
# Demonstrates a lifecycle node for an IMU sensor driver.
# ROS 2 Humble | Python 3.10+ | rclpy

import rclpy
from rclpy.lifecycle import LifecycleNode, TransitionCallbackReturn, State
from sensor_msgs.msg import Imu


class ImuLifecycleNode(LifecycleNode):
    """
    Lifecycle-managed IMU sensor node.

    State transitions:
      Unconfigured -> Inactive:  on_configure()  — open device, declare params
      Inactive -> Active:        on_activate()   — start publishing
      Active -> Inactive:        on_deactivate() — stop publishing, device idle
      Inactive -> Unconfigured:  on_cleanup()    — close device, release resources
      Any -> Finalized:          on_shutdown()   — emergency stop
    """

    def __init__(self, node_name: str = 'imu_driver') -> None:
        super().__init__(node_name)
        self._publisher = None
        self._timer = None
        self._device_handle = None  # Simulated hardware handle

    def on_configure(self, state: State) -> TransitionCallbackReturn:
        """
        Called when the 'configure' transition is triggered.
        Allocate resources: open hardware device, declare parameters, create publisher.
        Return SUCCESS to proceed to Inactive, FAILURE to stay in Unconfigured.
        """
        self.get_logger().info('Configuring IMU node...')

        # Declare parameters
        self.declare_parameter('imu_frame', 'imu_link')
        self.declare_parameter('publish_frequency', 200.0)  # Hz

        # Create publisher — allocated in Inactive state but not publishing yet
        self._publisher = self.create_lifecycle_publisher(
            msg_type=Imu,
            topic='/imu/data',
            qos_profile=10,
        )

        # Simulate hardware initialization
        self._device_handle = True  # In real code: open serial/I2C connection
        if not self._device_handle:
            self.get_logger().error('Failed to open IMU device')
            return TransitionCallbackReturn.FAILURE

        self.get_logger().info('IMU node configured successfully')
        return TransitionCallbackReturn.SUCCESS

    def on_activate(self, state: State) -> TransitionCallbackReturn:
        """
        Called when the 'activate' transition is triggered.
        Start the publishing timer. Node is now fully operational.
        """
        self.get_logger().info('Activating IMU node...')

        frequency = self.get_parameter('publish_frequency').value
        period = 1.0 / frequency

        self._timer = self.create_timer(period, self._publish_imu_data)

        # Activate the lifecycle publisher (required step)
        super().on_activate(state)

        self.get_logger().info(f'IMU node active at {frequency} Hz')
        return TransitionCallbackReturn.SUCCESS

    def on_deactivate(self, state: State) -> TransitionCallbackReturn:
        """
        Called when the 'deactivate' transition is triggered.
        Stop the timer but keep hardware connection open.
        """
        self.get_logger().info('Deactivating IMU node...')

        if self._timer is not None:
            self._timer.cancel()
            self._timer = None

        super().on_deactivate(state)
        self.get_logger().info('IMU node deactivated (device still connected)')
        return TransitionCallbackReturn.SUCCESS

    def on_cleanup(self, state: State) -> TransitionCallbackReturn:
        """
        Called when the 'cleanup' transition is triggered.
        Release all resources. Node returns to Unconfigured.
        """
        self.get_logger().info('Cleaning up IMU node...')
        self._publisher = None
        self._device_handle = None  # In real code: close hardware connection
        return TransitionCallbackReturn.SUCCESS

    def on_shutdown(self, state: State) -> TransitionCallbackReturn:
        """Emergency shutdown from any state."""
        self.get_logger().info('Shutting down IMU node')
        return TransitionCallbackReturn.SUCCESS

    def _publish_imu_data(self) -> None:
        """Timer callback: reads IMU hardware and publishes a sensor_msgs/Imu message."""
        if not self._publisher or not self._publisher.is_activated:
            return

        msg = Imu()
        msg.header.stamp = self.get_clock().now().to_msg()
        msg.header.frame_id = self.get_parameter('imu_frame').value

        # In a real driver, read actual sensor registers here.
        # Placeholder: zero acceleration and angular velocity
        msg.linear_acceleration.x = 0.0
        msg.linear_acceleration.y = 0.0
        msg.linear_acceleration.z = 9.81  # Gravity

        self._publisher.publish(msg)


def main(args=None) -> None:
    rclpy.init(args=args)
    node = ImuLifecycleNode()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## 3.6 Package Structure and Build System

ROS 2 software is organized into **packages** — the fundamental unit of build, distribution, and dependency management. Every package contains source code, resource files (meshes, configuration, launch files), and a `package.xml` manifest that declares the package's metadata and dependencies.

### 3.6.1 Package Types

ROS 2 supports three package build types, declared in `package.xml`:

**ament_cmake** — For C++ packages (and mixed C++/Python packages). Uses CMake as the underlying build system with ament's extensions for dependency management and installation conventions.

**ament_python** — For pure Python packages. Uses Python's `setuptools` under the hood and follows standard Python packaging conventions. Recommended for Python-only nodes.

**cmake** — For packages that use plain CMake without ament extensions. Rarely used in new ROS 2 code.

### 3.6.2 package.xml Structure

Every package has a `package.xml` file in its root directory that declares:

```xml
<?xml version="1.0"?>
<?xml-model href="http://download.ros.org/schema/package_format3.xsd"
            schematypens="http://www.w3.org/2001/XMLSchema"?>
<package format="3">
  <!-- Package identity -->
  <name>humanoid_arm_controller</name>
  <version>0.1.0</version>
  <description>
    Joint velocity and trajectory controller for a 3-DOF humanoid arm.
    Implements ROS 2 Humble lifecycle nodes and ros2_control integration.
  </description>

  <!-- Maintainer and license information (required for release) -->
  <maintainer email="faisal@example.com">Muhammad Faisal Laiq Siddiqui</maintainer>
  <license>Apache-2.0</license>

  <!-- Build-time dependencies: tools and libraries needed at build/test time -->
  <buildtool_depend>ament_python</buildtool_depend>

  <!-- Runtime dependencies: required when executing the node -->
  <depend>rclpy</depend>
  <depend>sensor_msgs</depend>
  <depend>control_msgs</depend>
  <depend>trajectory_msgs</depend>
  <depend>std_srvs</depend>
  <depend>rcl_interfaces</depend>

  <!-- Test dependencies: only needed during testing -->
  <test_depend>ament_copyright</test_depend>
  <test_depend>ament_flake8</test_depend>
  <test_depend>ament_pep257</test_depend>
  <test_depend>pytest</test_depend>

  <!-- Export: declares this package uses ament_python build type -->
  <export>
    <build_type>ament_python</build_type>
  </export>
</package>
```

### 3.6.4 The colcon Build Tool

**colcon** (collective construction) [7] is the build tool used for ROS 2 workspaces. Unlike ROS 1's `catkin_make`, colcon supports building packages in parallel and handles packages of different build types in the same workspace. Common colcon commands:

```bash
# Build all packages in the workspace
colcon build

# Build only specific packages (and their dependencies)
colcon build --packages-select humanoid_arm_controller

# Build with symlinks instead of copying files (faster iteration for Python packages)
colcon build --symlink-install

# Run tests for all packages
colcon test

# Show test results
colcon test-result --verbose
```

After building, source the workspace overlay to make the packages available:

```bash
source install/setup.bash
```

## 3.8 Chapter Summary

ROS 2 is the software layer that transforms a collection of sensors, actuators, and compute modules into a coordinated robot. Its architecture — peer-to-peer DDS communication without a central broker, Quality of Service policies that match delivery semantics to real-time requirements, and lifecycle management that enforces deterministic startup and shutdown — was engineered specifically to solve the limitations that made ROS 1 inadequate for production deployment.

The middleware foundation begins with the DDS standard: why it exists, why ROS 2 chose it over a custom transport, and how the RTPS protocol enables multi-host, multi-robot coordination without a rosmaster equivalent. Fast DDS and Cyclone DDS differ in latency profiles and configuration flexibility, but both implement the same QoS policy vocabulary — and choosing the right QoS profile for each topic (SENSOR_DATA for camera streams, SERVICES_DEFAULT for configuration commands) is the difference between a system that degrades gracefully and one that stalls unpredictably.

The DDS layer received detailed treatment: the RTPS wire protocol, vendor implementations (Fast DDS and Cyclone DDS), Quality of Service policies that give developers precise control over delivery semantics, and domain-based isolation for multi-robot systems.

We implemented all four communication primitives in Python:
- **Topics** for asynchronous streaming of joint velocity commands at 100 Hz
- **Services** for synchronous gripper control commands
- **Actions** for trajectory execution with intermediate feedback and cancellation support
- **Parameters** for runtime-configurable controller gains with validation callbacks

Lifecycle nodes were introduced as the mechanism for deterministic startup and shutdown in production systems — essential when a 44-DOF humanoid cannot afford unordered initialization or uncontrolled power-down sequences.

The build system section covered package types, `package.xml` structure, directory conventions for ament_python packages, and the `colcon` build tool. A complete, reproducible installation guide for ROS 2 Humble on Ubuntu 22.04 was provided.

Chapter 4 builds directly on this foundation: we move from ROS 2 communication architecture into the practical challenge of building humanoid robot control software with `rclpy`. We will implement multi-threaded executors, integrate with the `ros2_control` framework, author a complete URDF for a simplified humanoid arm, and connect motion planning through MoveIt 2.
